Hello,

Yes, this is exactly the topic I’ve been working on over the last days.

For me this is mainly a design question, not a missing function problem.

In a web context, long-running jobs simply don’t belong inside the web server process.
Whether it’s Hix, PHP, or any other web runtime: blocking calls, process execution with I/O, or long database work will always clash with request/response handling.

My conclusion so far:

short, reactive logic → web layer (Hix / PHP / JS)

long-running or stateful jobs → external service process

Node.js, Python or even a standalone Harbour EXE

communication via HTTP or WebSocket

This keeps the web layer responsive and avoids timeouts, while still allowing full control over longer jobs.

So in my view, Hix behaving this way actually confirms a clean separation of concerns, rather than being a limitation.

Thanks Bayron for sharing the example and for your continued support of Hix and FiveWin.

I think we actually agree on most fundamentals. The difference is mainly how we look at the workflow and responsibilities, not at Harbour’s capabilities.

To make this clearer, I’d like to put the two approaches side by side.

Hix-internal workflow (as in your example)
Browser
→ HTTP request
→ Hix executes PRG
→ optional file-based state (JSON)
→ immediate response to UI

In this model:

logic runs inside the web request
persistence is simulated via files
there is no job lifecycle beyond the request
the UI always drives execution

This works well as your code shows and clearly shows what Hix can do.

Our workflow (Hix + Harbour desktop program)
Browser / UI
→ Request to Hix
→ Hix triggers Harbour desktop EXE (ShellExecute + unique job id)
→ Harbour program runs independently
→ writes result/status (JSON or file)
→ UI fetches or polls the result when it is ready

Here:

Hix stays lightweight and reactive
long or blocking work runs outside the web process
state belongs to the job, not to the request
lifecycle, logging, recovery and cleanup are explicit

This is not about adding layers, but about separating roles early.

About Node.js
The only reason I mentioned Node.js earlier is that it is optional and portable, not because it is required.
A setup like:
Harbour (worker)
↔ WebSocket
↔ Node.js (SQL adapter)
↔ SQL database

is simply one interesting option, especially for SQL access:
no SQL libraries inside Harbour
no additional DLL complexity
clean isolation and well-maintained drivers

But to be very clear:
Node.js is not mandatory at all.

Hix can directly trigger and communicate with an existing Harbour desktop program.
No PHP, no additional framework, no extra runtime is required.

The core concern

What worries me from a product and long-term maintenance perspective is that once we start adding:

SQL libraries
threading
more dependencies
directly into the Hix process, we slowly turn a clean web layer into a hard-to-maintain monolith.

My intention is not to limit Hix, but to protect it from becoming overloaded with responsibilities.
