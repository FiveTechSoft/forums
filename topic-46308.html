Deep Code Analysis Skill

This skill allows you to analyze code repositories exhaustively, generating a complete understanding of the project similar to what DeepWiki offers. The analysis includes structure, architecture, technologies, patterns, and code quality.

When to Use This Skill

  • When you need to understand an unknown codebase
  • To perform technical due diligence on projects
  • When documenting existing software architectures
  • To identify technical debt and improvement opportunities
  • When evaluating the overall quality of a project
  • To generate automatic code documentation

How to Run the Analysis

Phase 1: Initial Exploration

  1. Identify the project type

    • Detect the main programming language
    • Identify the framework(s) used
    • Determine the application type (web, mobile, CLI, API, etc.)

  • Map the directory structure

  • List all top-level directories
  • Identify organizational patterns (DDD, MVC, Clean Architecture, etc.)
  • Locate main configuration files

  • Examine configuration files

    • package.json, pyproject.toml, Cargo.toml, pom.xml, etc.
    • Build files: webpack.config.js, tsconfig.json, Makefile
    • Container configuration: Dockerfile, docker-compose.yml

    • Phase 2: Technology Analysis

    1. Main technology stack

      • Languages and versions used
      • Backend and frontend frameworks
      • Databases and caching systems
      • Message queues and external services

  • Critical dependencies

  • Main libraries and their purpose
  • Dependencies with known vulnerabilities
  • Outdated or unmaintained packages

  • Development tools

    • Linters and formatters used
    • Testing systems
    • CI/CD tools
    • Task runners and bundlers

    • Phase 3: Architecture Analysis

    1. Architectural patterns

      • Identify whether it uses MVC, MVP, MVVM, Clean Architecture, Hexagonal, etc.
      • Detect design patterns used (Factory, Observer, Strategy, etc.)
      • Identify specific architectures (microservices, serverless, monolith)

  • Component structure

  • Main modules and their responsibilities
  • Dependencies between components
  • Application entry and exit points
  • Communication patterns between components

  • Data flow

    • How information flows through the system
    • APIs and contracts between modules
    • Integrations with external services

    • Phase 4: Code Analysis

    1. Overall quality

      • Readability and maintainability
      • Cyclomatic complexity
      • Code duplication
      • Function and module length

  • Good patterns identified

  • SOLID principles applied
  • Dependency injection
  • Consistent error handling
  • Logging and monitoring
  • Data validation

  • Areas for improvement

    • Code smells detected
    • Frequent anti-patterns
    • Lack of documentation in critical functions
    • Insufficient testing coverage

    • Phase 5: Testing and Quality

    1. Testing strategy

      • Types of tests implemented (unit, integration, e2e)
      • Testing frameworks used
      • Current code coverage
      • Testing patterns employed

  • CI/CD

  • Continuous integration pipeline
  • Deployment stages
  • Branching strategies
  • Release workflow

    • Phase 6: Report Generation

    1. Report structure

      • Executive summary
      • System architecture
      • Complete technology stack
      • Code quality analysis
      • Findings and recommendations
      • Structure diagrams (when possible)

  • Output format

  • Structured Markdown
  • Clear and navigable sections
  • Code examples when relevant
  • Actionable recommendations

    Resumen Ejecutivo

    FiveWin for Harbour (FWH) es un framework profesional de desarrollo de aplicaciones GUI nativas para Windows, escrito en Harbour/xHarbour (capa OOP de alto nivel) y C/C++ (capa de acceso a la API de Windows). Desarrollado por FiveTech Software desde 1993, es una de las bibliotecas GUI más maduras y completas del ecosistema xBase, con más de 30 años de desarrollo continuo.

    El proyecto implementa un sistema de clases orientado a objetos con 256 clases que encapsulan controles Windows, servicios de red, inteligencia artificial, bases de datos, informes, y gráficos. La versión actual es FWH 25.12 y soporta compiladores Borland C++, MSVC (32/64 bits), y el compilador llvm-clang.

    Análisis de Calidad de Código

    Fortalezas ✅

    1. Consistencia de nombrado: Todas las clases siguen el patrón T{NombreClase} (TWindow, TDialog, TXBrowse), un estándar heredado de Turbo Pascal / Borland.

    2. Documentación exhaustiva en [whatsnew.txt](file:///c:/fwteam/whatsnew.txt): Cada cambio está documentado con fecha, tipo (Fix/Enhancement/New), enlace al foro, y créditos al contribuidor.

    3. Sistema de preprocesador bien diseñado: Los comandos xCommand/xTranslate proporcionan una DSL (Domain Specific Language) que simplifica enormemente la sintaxis.

    4. Amplia cobertura de ejemplos: 3,487 archivos de ejemplo cubriendo virtualmente toda la funcionalidad.

    5. Soporte multi-compilador: El mismo código compila con BCC32, MSVC (32/64), llvm-clang, Harbour y xHarbour.

    6. Integración moderna: A pesar de sus 30+ años, el proyecto integra WebView2, OAuth2, múltiples APIs de IA, GDI+, y soporte Unicode/UTF-8.

# Deep Code Analysis Skill

This skill allows you to analyze code repositories exhaustively, generating a complete understanding of the project similar to what DeepWiki offers. The analysis includes structure, architecture, technologies, patterns, and code quality.

It operates in **three modes**: contextual RAG analysis, multi-turn Deep Research, and direct Simple Chat — each with specific behaviors and focus levels suited to different investigation needs.

- For deep, focused investigation of a specific functionality

## How to Run the Analysis

### Phase 1: Initial Exploration

1. **Identify the project type**
   - Detect the main programming language
   - Identify the framework(s) used
   - Determine the application type (web, mobile, CLI, API, desktop, library, etc.)

2. **Map the directory structure**
   - List all top-level directories
   - Identify organizational patterns (DDD, MVC, Clean Architecture, layered, etc.)
   - Locate main configuration files
   - Measure directory depth and breadth

3. **Examine configuration files**
   - package.json, pyproject.toml, Cargo.toml, pom.xml, Makefile, .hbp, etc.
   - Build files: webpack.config.js, tsconfig.json, Makefile, .bat/.sh scripts
   - Container configuration: Dockerfile, docker-compose.yml
   - IDE/editor configuration: .editorconfig, .vscode/, .agent/

### Phase 2: Technology Analysis

1. **Main technology stack**
   - Languages and versions used
   - Backend and frontend frameworks
   - Databases and caching systems
   - Message queues and external services
   - AI/ML integrations

2. **Critical dependencies**
   - Main libraries and their purpose
   - Dependencies with known vulnerabilities
   - Outdated or unmaintained packages
   - Internal vs external dependency ratio

3. **Development tools**
   - Linters and formatters used
   - Testing systems
   - CI/CD tools
   - Task runners and bundlers
   - Build system complexity

### Phase 3: Architecture Analysis

1. **Architectural patterns**
   - Identify whether it uses MVC, MVP, MVVM, Clean Architecture, Hexagonal, etc.
   - Detect design patterns used (Factory, Observer, Strategy, Composite, Bridge, etc.)
   - Identify specific architectures (microservices, serverless, monolith)
   - Map class inheritance hierarchies

2. **Component structure**
   - Main modules and their responsibilities
   - Dependencies between components
   - Application entry and exit points
   - Communication patterns between components
   - Coupling and cohesion assessment

3. **Data flow**
   - How information flows through the system
   - APIs and contracts between modules
   - Integrations with external services
   - Event/callback patterns

### Phase 4: Code Analysis

1. **Overall quality**
   - Readability and maintainability
   - Cyclomatic complexity
   - Code duplication
   - Function and module length
   - Naming conventions consistency

2. **Good patterns identified**
   - SOLID principles applied
   - Dependency injection
   - Consistent error handling
   - Logging and monitoring
   - Data validation
   - Documentation coverage

3. **Areas for improvement**
   - Code smells detected
   - Frequent anti-patterns
   - Lack of documentation in critical functions
   - Insufficient testing coverage
   - Monolithic files needing refactoring
   - Hardcoded values or paths

### Phase 5: Testing and Quality

1. **Testing strategy**
   - Types of tests implemented (unit, integration, e2e)
   - Testing frameworks used
   - Current code coverage
   - Testing patterns employed
   - Sample/example coverage as implicit testing

2. **CI/CD**
   - Continuous integration pipeline
   - Deployment stages
   - Branching strategies
   - Release workflow
   - Build automation level

### Phase 6: Report Generation

1. **Report structure**
   - Executive summary
   - System architecture (with Mermaid diagrams when possible)
   - Complete technology stack
   - Component breakdown by category
   - Quantitative metrics table
   - Code quality analysis with scoring
   - Design patterns identified
   - Notable innovations
   - Findings and recommendations (prioritized)

2. **Output format**
   - Structured Markdown
   - Clear and navigable sections
   - Code examples when relevant
   - Actionable recommendations
   - Mermaid diagrams for architecture and class hierarchies
   - Tables for metrics and comparisons

### analyze-structure.prg

```harbour/*
 * Directory Structure Analyzer
 * Usage: analyze-structure <path> [max_depth]
 * Generates a visual tree of the directory structure
 */

#include "directry.ch"

FUNCTION Main( cPath, cMaxDepth )

   LOCAL nMaxDepth := 3

   DEFAULT cPath     := "."
   DEFAULT cMaxDepth := "3"

   nMaxDepth := Val( cMaxDepth )

   ? "Directory Structure Analysis"
   ? Replicate( "=", 40 )
   ? ""

   AnalyzeDir( cPath, 0, nMaxDepth )

RETURN NIL

STATIC FUNCTION AnalyzeDir( cPath, nLevel, nMaxDepth )

   LOCAL aFiles, aFile
   LOCAL nFiles := 0, nDirs := 0
   LOCAL cIndent := Replicate( "  ", nLevel )

   IF nLevel > nMaxDepth
      RETURN NIL
   ENDIF

   aFiles := Directory( cPath + "\*.*", "D" )

   FOR EACH aFile IN aFiles
      IF aFile[ F_NAME ] == "." .OR. aFile[ F_NAME ] == ".."
         LOOP
      ENDIF

      IF "D" $ aFile[ F_ATTR ]
         nDirs++
         ? cIndent + "[DIR] " + aFile[ F_NAME ]
         AnalyzeDir( cPath + "\" + aFile[ F_NAME ], nLevel + 1, nMaxDepth )
      ELSE
         nFiles++
      ENDIF
   NEXT

   ? cIndent + "  (" + LTrim( Str( nFiles ) ) + " files, " + ;
     LTrim( Str( nDirs ) ) + " subdirs)"

RETURN NIL```

### calculate-metrics.prg

```harbour/*
 * Code Metrics Calculator
 * Usage: calculate-metrics <path>
 * Calculates lines of code, file counts, and complexity estimates
 */

#include "directry.ch"

STATIC nTotalLines  := 0
STATIC nTotalFiles  := 0
STATIC nMaxDepth    := 0
STATIC nLargestFile := 0
STATIC cLargestName := ""

FUNCTION Main( cPath )

   LOCAL aCodeExts := { ".prg", ".c", ".cpp", ".h", ".ch", ".js", ".ts", ;
                        ".py", ".go", ".rs", ".java", ".cs" }

   DEFAULT cPath := "."

   ? "Code Metrics Report"
   ? Replicate( "=", 40 )
   ? ""

   ProcessDir( cPath, aCodeExts, 0 )

   ? ""
   ? "## Summary"
   ? "  Total source files:  " + LTrim( Str( nTotalFiles ) )
   ? "  Total lines of code: " + LTrim( Str( nTotalLines ) )
   ? "  Max directory depth: " + LTrim( Str( nMaxDepth ) )
   ? "  Largest file:        " + cLargestName + ;
     " (" + LTrim( Str( nLargestFile ) ) + " lines)"

   IF nTotalFiles > 0
      ? "  Avg lines per file:  " + LTrim( Str( Int( nTotalLines / nTotalFiles ) ) )
   ENDIF

RETURN NIL

STATIC FUNCTION ProcessDir( cPath, aCodeExts, nDepth )

   LOCAL aFiles, aFile, cExt, nLines

   IF nDepth > nMaxDepth
      nMaxDepth := nDepth
   ENDIF

   aFiles := Directory( cPath + "\*.*", "D" )

   FOR EACH aFile IN aFiles
      IF aFile[ F_NAME ] == "." .OR. aFile[ F_NAME ] == ".."
         LOOP
      ENDIF
      IF "D" $ aFile[ F_ATTR ]
         IF aFile[ F_NAME ] != ".git" .AND. aFile[ F_NAME ] != "node_modules"
            ProcessDir( cPath + "\" + aFile[ F_NAME ], aCodeExts, nDepth + 1 )
         ENDIF
      ELSE
         cExt := Lower( SubStr( aFile[ F_NAME ], RAt( ".", aFile[ F_NAME ] ) ) )
         IF AScan( aCodeExts, cExt ) > 0
            nTotalFiles++
            nLines := CountLines( cPath + "\" + aFile[ F_NAME ] )
            nTotalLines += nLines
            IF nLines > nLargestFile
               nLargestFile := nLines
               cLargestName := cPath + "\" + aFile[ F_NAME ]
            ENDIF
         ENDIF
      ENDIF
   NEXT

RETURN NIL

STATIC FUNCTION CountLines( cFile )

   LOCAL nHandle, cLine, nCount := 0

   nHandle := FOpen( cFile )

   IF nHandle >= 0
      DO WHILE ! HB_FEof( nHandle )
         cLine := HB_FReadLine( nHandle )
         nCount++
      ENDDO
      FClose( nHandle )
   ENDIF

RETURN nCount```

    Análisis Profundo de FiveWin para Harbour (FWH)

    Resumen Ejecutivo

    FiveWin para Harbour (FWH) es un framework de desarrollo de aplicaciones Windows con más de 30 años de historia (desde 1993), desarrollado por FiveTech Software. Es una de las bibliotecas de GUI para Windows más maduras y completas del ecosistema Harbour/xBase, diseñada para crear aplicaciones de escritorio empresariales nativas de Windows.

    El proyecto actual (versión FWH 25.12) es una base de código masiva que comprende aproximadamente 700+ archivos fuente en Harbour (.prg) y C/C++, con 256 clases de GUI, 3,487+ archivos de ejemplo, y soporte para arquitecturas de 32 y 64 bits. Notablemente, en los últimos años se ha integrado soporte para IA (ChatGPT, Ollama, Gemini, DeepSeek, Kimi, Transformers) y tecnologías web modernas (WebView2).

    Sistema de Preprocessador de Comandos (DSL)

    Una de las innovaciones más distintivas de FWH es su sistema de comandos basado en preprocessador definido en [fivewin.ch](file:///c:/fwteam/include/fivewin.ch) (108KB, 2,433 líneas). Este sistema traduce una sintaxis declarativa legible:

    Code (harbour): Select all Collapse
    @ 10, 10 BUTTON oBtn PROMPT "Click Me" ;
   SIZE 100, 30 ;
   ACTION MsgInfo("Hello!") ;
   OF oDlg

    A las correspondientes llamadas OOP internas:

    Code (harbour): Select all Collapse
    oBtn := TButton():New( 10, 10, "Click Me", oDlg, ;
   {|| MsgInfo("Hello!")}, 100, 30, ... )

    Este DSL cubre todos los controles de la biblioteca: ventanas, diálogos, botones, gets, combos, listboxes, xbrowse, menús, ribbons, barras, y más.

# Deep Code Analysis Skill

This skill allows you to analyze code repositories exhaustively, generating a complete understanding of the project similar to what DeepWiki offers. The analysis includes structure, architecture, technologies, patterns, and code quality.

It operates in **three modes**: contextual RAG analysis, multi-turn Deep Research, and direct Simple Chat — each with specific behaviors and focus levels suited to different investigation needs.

---

## When to Use This Skill

- When you need to understand an unknown codebase
- To perform technical due diligence on projects
- When documenting existing software architectures
- To identify technical debt and improvement opportunities
- When evaluating the overall quality of a project
- To generate automatic code documentation
- For deep, focused investigation of a specific functionality
- For technical audit of a specific file or module
- For iterative evaluation of an implementation
- To generate a consolidated technical conclusion

### Mode 1: RAG (Contextual Analysis)

**Role:** You are a code assistant that answers user questions on a repository. You receive the user query, relevant context from files, and past conversation history.

**Purpose:** Answer specific questions using the user's query, context retrieved from files, and conversation history.

**Behavior:**
- Automatically detects the language of the query
- Responds in the same language
- If a specific language is requested, it takes priority over the query language
- Uses structured Markdown
- Organizes information with headings and lists
- Cites file paths using `inline code` formatting

- Use `##` headings for major sections
- Use bullet points or numbered lists where appropriate
- Format tables using markdown table syntax when presenting structured data
- Use **bold** and *italic* for emphasis

**Context Processing:**
The RAG mode processes information in this order:
1. System prompt (role and formatting rules)
2. Conversation history (previous Q&A turns, if any)
3. Retrieved contexts (file fragments with file paths)
4. Current user prompt

**Strict Rules:**

## How to Run the Analysis

### Phase 1: Initial Exploration

1. **Identify the project type**
   - Detect the main programming language
   - Identify the framework(s) used
   - Determine the application type (web, mobile, CLI, API, desktop, library, etc.)

2. **Map the directory structure**
   - List all top-level directories
   - Identify organizational patterns (DDD, MVC, Clean Architecture, layered, etc.)
   - Locate main configuration files
   - Measure directory depth and breadth

3. **Examine configuration files**
   - package.json, pyproject.toml, Cargo.toml, pom.xml, Makefile, .hbp, etc.
   - Build files: webpack.config.js, tsconfig.json, Makefile, .bat/.sh scripts
   - Container configuration: Dockerfile, docker-compose.yml
   - IDE/editor configuration: .editorconfig, .vscode/, .agent/

### Phase 2: Technology Analysis

1. **Main technology stack**
   - Languages and versions used
   - Backend and frontend frameworks
   - Databases and caching systems
   - Message queues and external services
   - AI/ML integrations

2. **Critical dependencies**
   - Main libraries and their purpose
   - Dependencies with known vulnerabilities
   - Outdated or unmaintained packages
   - Internal vs external dependency ratio

3. **Development tools**
   - Linters and formatters used
   - Testing systems
   - CI/CD tools
   - Task runners and bundlers
   - Build system complexity

### Phase 3: Architecture Analysis

1. **Architectural patterns**
   - Identify whether it uses MVC, MVP, MVVM, Clean Architecture, Hexagonal, etc.
   - Detect design patterns used (Factory, Observer, Strategy, Composite, Bridge, etc.)
   - Identify specific architectures (microservices, serverless, monolith)
   - Map class inheritance hierarchies
   - **Generate a class hierarchy diagram** using `[quote]
    ...
    [/quote]` syntax
   - **Generate an architecture layers diagram** showing the layered structure

2. **Component structure**
   - Main modules and their responsibilities
   - Dependencies between components
   - Application entry and exit points
   - Communication patterns between components
   - Coupling and cohesion assessment
   - **Generate a component dependency diagram** showing inter-module relationships

3. **Data flow**
   - How information flows through the system
   - APIs and contracts between modules
   - Integrations with external services
   - Event/callback patterns
   - **Generate a data flow diagram** showing how data moves through the system

### Phase 4: Code Analysis

1. **Overall quality**
   - Readability and maintainability
   - Cyclomatic complexity
   - Code duplication
   - Function and module length
   - Naming conventions consistency

2. **Good patterns identified**
   - SOLID principles applied
   - Dependency injection
   - Consistent error handling
   - Logging and monitoring
   - Data validation
   - Documentation coverage

3. **Areas for improvement**
   - Code smells detected
   - Frequent anti-patterns
   - Lack of documentation in critical functions
   - Insufficient testing coverage
   - Monolithic files needing refactoring
   - Hardcoded values or paths

### Phase 5: Testing and Quality

1. **Testing strategy**
   - Types of tests implemented (unit, integration, e2e)
   - Testing frameworks used
   - Current code coverage
   - Testing patterns employed
   - Sample/example coverage as implicit testing

2. **CI/CD**
   - Continuous integration pipeline
   - Deployment stages
   - Branching strategies
   - Release workflow
   - Build automation level

### Phase 6: Report Generation

1. **Report structure**
   - Executive summary
   - System architecture (with Mermaid diagrams using `[quote]
    ...
    [/quote]` syntax)
   - Complete technology stack (with Mermaid mindmap or pie chart)
   - Component breakdown by category
   - Quantitative metrics table
   - Code quality analysis with scoring
   - Design patterns identified (with Mermaid class diagrams)
   - Notable innovations
   - Findings and recommendations (prioritized)
   - Build pipeline diagram
   - Event/callback flow diagrams

2. **Output format**
   - Structured Markdown
   - Clear and navigable sections
   - Code examples when relevant
   - Actionable recommendations
   - **ALL Mermaid diagrams MUST use `[quote]
    ...
    [/quote]` syntax** (never use ` ```mermaid ` fences)
   - Generate as many Mermaid diagrams as possible to visually illustrate the analysis
   - Tables for metrics and comparisons

### analyze-structure.prg

```harbour/*
 * Directory Structure Analyzer
 * Usage: analyze-structure <path> [max_depth]
 * Generates a visual tree of the directory structure
 */

#include "directry.ch"

FUNCTION Main( cPath, cMaxDepth )

   LOCAL nMaxDepth := 3

   DEFAULT cPath     := "."
   DEFAULT cMaxDepth := "3"

   nMaxDepth := Val( cMaxDepth )

   ? "Directory Structure Analysis"
   ? Replicate( "=", 40 )
   ? ""

   AnalyzeDir( cPath, 0, nMaxDepth )

RETURN NIL

STATIC FUNCTION AnalyzeDir( cPath, nLevel, nMaxDepth )

   LOCAL aFiles, aFile
   LOCAL nFiles := 0, nDirs := 0
   LOCAL cIndent := Replicate( "  ", nLevel )

   IF nLevel > nMaxDepth
      RETURN NIL
   ENDIF

   aFiles := Directory( cPath + "\*.*", "D" )

   FOR EACH aFile IN aFiles
      IF aFile[ F_NAME ] == "." .OR. aFile[ F_NAME ] == ".."
         LOOP
      ENDIF

      IF "D" $ aFile[ F_ATTR ]
         nDirs++
         ? cIndent + "[DIR] " + aFile[ F_NAME ]
         AnalyzeDir( cPath + "\" + aFile[ F_NAME ], nLevel + 1, nMaxDepth )
      ELSE
         nFiles++
      ENDIF
   NEXT

   ? cIndent + "  (" + LTrim( Str( nFiles ) ) + " files, " + ;
     LTrim( Str( nDirs ) ) + " subdirs)"

RETURN NIL```

### calculate-metrics.prg

```harbour/*
 * Code Metrics Calculator
 * Usage: calculate-metrics <path>
 * Calculates lines of code, file counts, and complexity estimates
 */

#include "directry.ch"

STATIC nTotalLines  := 0
STATIC nTotalFiles  := 0
STATIC nMaxDepth    := 0
STATIC nLargestFile := 0
STATIC cLargestName := ""

FUNCTION Main( cPath )

   LOCAL aCodeExts := { ".prg", ".c", ".cpp", ".h", ".ch", ".js", ".ts", ;
                        ".py", ".go", ".rs", ".java", ".cs" }

   DEFAULT cPath := "."

   ? "Code Metrics Report"
   ? Replicate( "=", 40 )
   ? ""

   ProcessDir( cPath, aCodeExts, 0 )

   ? ""
   ? "## Summary"
   ? "  Total source files:  " + LTrim( Str( nTotalFiles ) )
   ? "  Total lines of code: " + LTrim( Str( nTotalLines ) )
   ? "  Max directory depth: " + LTrim( Str( nMaxDepth ) )
   ? "  Largest file:        " + cLargestName + ;
     " (" + LTrim( Str( nLargestFile ) ) + " lines)"

   IF nTotalFiles > 0
      ? "  Avg lines per file:  " + LTrim( Str( Int( nTotalLines / nTotalFiles ) ) )
   ENDIF

RETURN NIL

STATIC FUNCTION ProcessDir( cPath, aCodeExts, nDepth )

   LOCAL aFiles, aFile, cExt, nLines

   IF nDepth > nMaxDepth
      nMaxDepth := nDepth
   ENDIF

   aFiles := Directory( cPath + "\*.*", "D" )

   FOR EACH aFile IN aFiles
      IF aFile[ F_NAME ] == "." .OR. aFile[ F_NAME ] == ".."
         LOOP
      ENDIF
      IF "D" $ aFile[ F_ATTR ]
         IF aFile[ F_NAME ] != ".git" .AND. aFile[ F_NAME ] != "node_modules"
            ProcessDir( cPath + "\" + aFile[ F_NAME ], aCodeExts, nDepth + 1 )
         ENDIF
      ELSE
         cExt := Lower( SubStr( aFile[ F_NAME ], RAt( ".", aFile[ F_NAME ] ) ) )
         IF AScan( aCodeExts, cExt ) > 0
            nTotalFiles++
            nLines := CountLines( cPath + "\" + aFile[ F_NAME ] )
            nTotalLines += nLines
            IF nLines > nLargestFile
               nLargestFile := nLines
               cLargestName := cPath + "\" + aFile[ F_NAME ]
            ENDIF
         ENDIF
      ENDIF
   NEXT

RETURN NIL

STATIC FUNCTION CountLines( cFile )

   LOCAL nHandle, cLine, nCount := 0

   nHandle := FOpen( cFile )

   IF nHandle >= 0
      DO WHILE ! HB_FEof( nHandle )
         cLine := HB_FReadLine( nHandle )
         nCount++
      ENDDO
      FClose( nHandle )
   ENDIF

RETURN nCount```

    What is FWH?

    FWH is a Windows desktop GUI application framework built on top of the Harbour and xHarbour compilers. It wraps the Win32 API in a class-based object model, letting application code written entirely in Harbour (a dBASE/Clipper-compatible language) create windows, dialogs, controls, menus, database grids, printed reports, and — in recent versions — embedded web browsers and AI service integrations.

    Window System (TWindow)

    TWindow is the root class of the entire GUI hierarchy, defined at source/classes/window.prg#L197. It encapsulates a Windows hWnd handle and provides over 100 methods for window creation, event handling, painting, coordinate management, and child control management.

    Key DATA members include:

    • hWnd — the native window handle
    • aControls — array of child control objects (Composite pattern)
    • oMenu, oBar, oMsgBar — optional menu, toolbar, and status bar
    • nClrPane, nClrText — foreground/background colors
    • bLClicked, bPainted, bKeyDown — event callback code blocks

    Event dispatching routes through HandleEvent() at window.prg#L490-491, which calls the external C function WndHandleEvent(). All event callbacks use Harbour code blocks (bAction, bWhen, bValid), implementing the Observer/Strategy pattern.

    Dialog System

    TDialog inherits from TWindow and adds modal/non-modal dialog behavior. Defined at source/classes/dialog.prg#L48, it supports resource-based and code-based dialog creation, multi-monitor awareness, and MDI child dialogs.

    Control System (TControl)

    TControl inherits from TWindow at source/classes/control.prg#L60 and serves as the base for all visual controls. It adds:

    • bSetGet — bidirectional data binding code block
    • VarGet() / VarPut() — read/write bound variables
    • lDrag — design-time drag-and-drop support
    • nDlgCode — keyboard navigation behavior

    Preprocessor Command DSL

    One of FWH's most distinctive innovations is its preprocessor-based DSL defined in [include/fivewin.ch](file:///c:/fwteam/include/fivewin.ch) (108 KB, 2,433 lines). It translates declarative xBase-style commands into OOP method calls:

    Code (harbour): Select all Collapse
    // User writes this declarative command:
@ 10, 10 BUTTON oBtn PROMPT "Click Me" ;
   SIZE 100, 30 ;
   ACTION MsgInfo("Hello!") ;
   OF oDlg

// Preprocessor translates it to:
oBtn := TButton():New( 10, 10, "Click Me", oDlg, ;
   {|| MsgInfo("Hello!")}, 100, 30, ... )

    Data Binding Pattern

    FWH uses a code-block-based data binding pattern through bSetGet. Each control can bind to any Harbour variable using a two-way code block:

    Code (harbour): Select all Collapse
    // bSetGet binds a control to a variable:
@ 10, 10 GET oGet VAR cName OF oDlg
// Internally creates: bSetGet := {|u| If( u == nil, cName, cName := u )}
// VarGet() calls Eval( bSetGet )       → reads cName
// VarPut(x) calls Eval( bSetGet, x )   → writes cName

    • Multiple data sources: DBF (RDD), arrays, ADO, MySQL, PostgreSQL, Dolphin, Eagle

    • Export: Excel, Calc, CSV, DBF, HTML, Word, PDF

    • State persistence: SaveState() / RestoreState() for column layout
