Estimado Quim,



El archivo null.map que mencionas parece ser un archivo de mapa generado por el enlazador (linker) durante el proceso de compilación con Harbour 3.2.0dev (r2502110321) y BCC 7.70 al usar hbmk2. Estos archivos .map suelen contener información sobre las secciones de memoria del ejecutable resultante, como las que indicas (_TEXT, _DATA, _BSS, _TLS), que corresponden al código, datos inicializados, datos no inicializados y datos específicos de subprocesos, respectivamente. Su presencia no afecta el funcionamiento del ejecutable, como bien notas, pero su aparición repentina tras la actualización puede ser una peculiaridad del entorno o una configuración implícita en la nueva versión de Harbour o en cómo hbmk2 interactúa con el enlazador de BCC.



En proyectos con hbmk2, este comportamiento podría estar relacionado con cómo el script de construcción invoca al enlazador de Borland (BCC), que por defecto puede estar generando este archivo de mapa incluso si no se especifica explícitamente. En contraste, tus proyectos con FWH que usan makefile no muestran este archivo, probablemente porque el proceso de enlace está configurado de manera diferente, omitiendo la generación del archivo de mapa o redirigiéndolo a otro nombre o ubicación.



Aquí hay algunas posibles razones y pasos para investigar o resolver esta "curiosidad":



Cambio en Harbour o hbmk2: La versión específica de Harbour 3.2.0dev (r2502110321) podría haber introducido un ajuste en hbmk2 que activa la generación de archivos de mapa por defecto al usar BCC 7.70. Esto no parece estar documentado como un cambio importante, pero podría ser un efecto secundario de alguna actualización en el manejo de opciones del enlazador.



Configuración del enlazador de BCC: El enlazador de Borland (ilink32.exe o similar) genera archivos .map si se le pasa la opción -M (o si esta se activa implícitamente). El nombre null.map sugiere que no se especificó un nombre explícito para el archivo de mapa, y el enlazador usa un valor predeterminado (null). Puedes verificar esto ejecutando hbmk2 con la opción -trace para inspeccionar los comandos exactos que se pasan al enlazador.



Diferencia entre hbmk2 y makefiles: En tus proyectos FWH con makefile, es probable que el proceso de enlace esté configurado para no generar el archivo .map (por ejemplo, omitiendo la opción -M) o para nombrarlo de acuerdo con el proyecto (como <nombre>.map). Con hbmk2, si no se especifica una opción explícita para controlar esto, el comportamiento por defecto del enlazador podría ser el culpable.



¿Qué puedes hacer?

Confirmar el origen: Ejecuta tu comando hbmk2 con -trace (por ejemplo, hbmk2 tu_proyecto.hbp -trace) y revisa la salida para ver si el enlazador está recibiendo una instrucción que genera el archivo .map. Busca algo como -M o una opción similar en la línea de comando del enlazador.



Desactivar la generación del archivo: Si no necesitas el archivo .map, puedes intentar pasarle una opción al enlazador a través de hbmk2. Por ejemplo, agrega -ldflag=-M- a tu comando o archivo .hbp para intentar desactivar la generación del mapa (aunque la sintaxis exacta depende del enlazador de BCC). Consulta la documentación de ilink32.exe para confirmar las opciones disponibles.



Especificar un nombre personalizado: Si prefieres que el archivo .map tenga un nombre útil en lugar de null.map, puedes usar la opción -map en hbmk2 para definirlo explícitamente (por ejemplo, -map=mi_proyecto.map).



Revisar cambios en Harbour: Dado que esto comenzó tras actualizar a r2502110321, podrías revisar el registro de cambios (ChangeLog) de Harbour en el repositorio oficial para ver si hay menciones sobre modificaciones en hbmk2 o en el soporte para BCC que expliquen este comportamiento.



¿A alguien más le ha pasado?



No hay reportes específicos y masivos sobre este fenómeno con Harbour 3.2.0dev (r2502110321) y BCC 7.70 en la comunidad, pero es posible que otros usuarios de BCC y hbmk2 hayan notado algo similar y no lo hayan considerado un problema digno de mención, especialmente porque, como dices, no afecta el resultado final. La aparición de null.map parece más una curiosidad técnica que un error grave.



Si decides investigarlo más a fondo y encuentras la causa exacta (por ejemplo, una opción predeterminada en hbmk2), podrías compartirlo con la comunidad de Harbour para documentarlo. Por ahora, si no te molesta su presencia, puedes ignorarlo sin preocupaciones, ya que no tiene impacto funcional.

La respuesta correcta es añadir el flag -map al fichero de proyecto hbp


Code: Select all Collapse
</s><i>
</i>#-inc enable incremental build mode
-n -q0 -static -map
<e>


Si no se especifica map, se crea por defecto



De todas formas, con las respuestas siempre se aprende, por ejemplo, es una buena práctica incluir en la llamada hbmk2 -trace



Así podemos comprobar exactamente lo que está haciendo entre bambalinas la super utilidad hbmk2 de harbour
