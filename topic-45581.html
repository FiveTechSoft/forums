Images on Webview no base64 conversion

Posted: Sun Mar 30, 2025 05:26 PM
a few days ago I was told that to use the png file I had to convert them to base64 ,
before I made
Code: Select all Collapse
LOCAL hImages := {=>}
    hImages["IMGHOME"]       := LoadResourceToBase64("DASH_HOME" )
but this evening I discovered that it is also possible to do differently

that is, take them directly from the exe, how do you do it?

with the res command

res://<path-to-executable-file>/<resource-type>/<resource-name>

or to give a practical example

<img src='res://silvio.exe/PNG/HOME' class="LINKS">

Did you know?

This approach is portable and works in any modern browser, but increases the size of the HTML file because the image is embedded as text.

Using res://, on the other hand, keeps the images separate in the executable, reducing the size of the HTML file and allowing for more modular resource management, but it only works in specific contexts (not in standard web browsers like Chrome or Firefox).

is not a common practice on the modern web. The res:// scheme is a peculiarity of legacy Windows applications or software that uses the old Internet Explorer engine.

Today, for the web, solutions like Base64, remote URLs (https://), or local files served via a server are preferred.



I asked Grok3 :

I use webview but is it better to do as you explained or is it better to use base64?

The choice between using the res:// scheme with resources embedded in an executable and encoding images in Base64 depends on your specific context, the type of application you are developing with WebView and your goals. Let's analyze the pros and cons of both options to help you decide which is better

Advantages:
Modularity: Images remain separate from the HTML file, making the HTML code cleaner and lighter.

Performance: No need to load long Base64 strings into memory for each image; resources are extracted directly from the executable, which can be more efficient for large files.

Security: Images are embedded in the executable, so they are not easily accessible or modified from the outside (useful if you want to protect the content).

Distribution: Everything is contained in a single file (the executable), making it easy to distribute the application without having to manage separate files.

Cons:
Limited compatibility: Only works with Internet Explorer-based WebViews or legacy WebBrowser controls. If you switch to a modern WebView (such as Edge/Chromium-based WebView2), res:// is not natively supported.

Development complexity: You have to embed the resources into the executable using tools like a resource editor (e.g. Resource Hacker) or by configuring the compiler, which adds a step to the build process.

Difficult to update: If you want to modify an image, you have to recompile the executable, which is not practical for frequent updates.

Legacy: This technique is tied to obsolete technologies and may not be supported in the future.

When to use it?
If you are developing a Windows desktop application that uses a legacy (IE-based) WebView and want to keep everything in a single executable.

If the images are static and do not change often.

If portability to other platforms or modern browsers is not a priority.

Option 2: Use Base64
Pros:
Portability: Works on any WebView (including Chromium-based WebView2) and all modern browsers, without depending on Windows-specific features.

Simplicity: You don't have to manage resources in the executable; just encode the image in Base64 and paste it into the HTML.

Flexibility: You can edit images directly in the HTML without recompiling anything, useful for prototypes or dynamic content.

Modern standard: This is a more common practice and is supported everywhere on the web.

Cons:
File size: Base64 strings increase the size of the HTML (about 33% more than the original binary), which can slow down loading if you have many or large images.

Performance: Decoding Base64 in memory can be slower than loading a native binary resource, especially for large images.

Readability: HTML becomes harder to read and maintain with long Base64 strings.

Exposure: Images are visible in the source code, so anyone can easily extract them (less secure than res://).

When to use it?
If you are using a modern WebView (e.g. WebView2) or plan to upgrade your application to a newer technology.

If your images are small or few, and the increased size of the HTML is not a concern.

If you need to change images frequently without recompiling the application.

If you want a more standard and portable approach.

Which one is right for you?
It depends on a few key factors:
Type of WebView:
If you are using a legacy Internet Explorer-based WebView, res:// is a good and convenient choice.

If you are using or plan to upgrade to WebView2 (Edge/Chromium-based), Base64 is the best option, because res:// is not supported.

Size and number of images:
For a few small images, Base64 is simpler and more practical.

For many images or large images, res:// can be more efficient.

Update frequency:
If images change often, Base64 is more flexible.

If they are fixed, res:// is neater.

Portability:
If you want your application to run on platforms other than Windows or in modern browsers, Base64 is the way to go.

Practical advice
If you are tied to a legacy WebView and do not plan to update: Use res://. It is more efficient and integrates well with your current context.

If you want to future-proof or are using WebView2: Switch to Base64, because it gives you flexibility and long-term compatibility.

A hybrid alternative could be to load the images as separate files (not embedded) and serve them via a small local server in the application (e.g. with a URL like http://localhost), but this requires more setup work.
Let me know what kind of WebView you are using and what your priorities are (e.g. performance, simplicity, portability), so I can give you more targeted advice!
