Experience Report: Two Months with Our Local AI System
Posted: Sat Apr 05, 2025 06:16 AM

Hello friends,

Two months after launching our local AI setup – based on Ollama running on Ubuntu with a RAG (Retrieval-Augmented Generation) extension – it’s clear: our daily work with documents has significantly changed for the better.

Our Document Management System (DMS) follows a deliberately simple structure: HTML files, JSON metadata, and organized folder hierarchies. Nothing fancy or overly complex. The key difference now is the new AI integration: all contents are automatically transferred into a vector database (Chroma), where they’re indexed and stored with context. The locally installed AI can access this data – and provides clear, accurate answers.

What has changed in practice:

The DMS became a knowledge source: Instead of just storing documents, the AI “understands” their content and can respond meaningfully.

Context-based answers: The AI doesn’t just look for keywords – it intelligently searches across documents and combines relevant information.

Fully automated integration: Every new document automatically becomes part of the knowledge network – without extra effort.

High-quality responses: The answers are precise, easy to understand, and far superior to traditional full-text search.

100% local operation: No cloud, no API fees, no internet dependency – and full control over data privacy.

A surprising bonus:

What really stands out is the practical side effect: Everyone in the company – whether at the front desk, in the kitchen, or in maintenance – contributes to the AI knowledge base simply by writing into the DMS. Without even realizing it. It’s what we’d call effortless knowledge management.

For developers:

Integration is refreshingly simple. You send a text request via the local HTTP API and receive a JSON response. That’s it. Existing applications can easily tap into the system without major changes – but with a huge gain in functionality.

Conclusion:

For us, combining a local DMS with AI is far from a hype. It’s a practical, forward-thinking step – especially for small businesses that want to actively use their internal knowledge instead of just storing it.

Best regards,

Otto

File Details
How does it work to install a virtual box for Windows on a Linux PC? I need Windows Server.

Sure, I'll explain it step by step – so you want to run a virtual machine with Windows Server on a Linux computer (e.g., Ubuntu), for testing or as an RDP target. Here's the guide:

🧰 Requirements

A Linux PC with sufficient RAM (at least 8 GB recommended)

Installed Linux operating system (e.g., Ubuntu, Debian, Fedora …)

Windows Server ISO file (e.g., Windows Server 2019, 2022)

VirtualBox + Extension Pack

🔧 Step 1: Install VirtualBox
For Ubuntu/Debian:

bash
Kopieren
Bearbeiten
sudo apt update
sudo apt install virtualbox
For Fedora/RHEL-based:

bash
Kopieren
Bearbeiten
sudo dnf install virtualbox
Alternatively, you can get the latest version directly from Oracle’s website.

🧩 Step 2: Install Extension Pack
The Extension Pack is needed for USB 2.0/3.0, RDP, etc.

Download here: https://www.virtualbox.org/wiki/Downloads

Then install:

bash
Kopieren
Bearbeiten
sudo VBoxManage extpack install <path-to-.vbox-extpack-file>
Or via the GUI: Open VirtualBox → File → Preferences → Extensions.

💿 Step 3: Create a VM for Windows Server

Start VirtualBox

Create new VM:

Name: e.g., WinServer2022

Type: Microsoft Windows

Version: Windows 2022 (64-bit), or depending on ISO

RAM: e.g., 4096 MB (or more, depending on system)

Hard disk: new virtual disk (e.g., 50 GB, VDI, dynamically allocated)

Then: Select VM → Settings → Storage → Insert ISO file as virtual CD/DVD

▶️ Step 4: Install Windows Server

Start the VM

Install Windows Server as you would on a real PC

At the end, activate Windows Server, set up RDP, etc.

🔐 Optional: Set up RDP access to the VM
If you want to access the VM via RDP:

Enable RDP in Windows Server

Set the network mode in VM settings to Bridged Adapter (so the VM gets its own IP address on the network)

Question: What is the minimum requirement for the customer's system to work reasonably well?

Answer: For reasonably smooth operation of a VirtualBox VM with Windows Server (e.g., for testing or RDP access), the host system (i.e., the customer’s Linux PC) should meet at least the following requirements:

✅ Minimum Requirements for Host PC

Component Recommendation for productive use
CPU Quad-core processor, e.g., Intel i5 / AMD Ryzen 5 or better
RAM At least 16 GB (so that 4–8 GB can be allocated to the VM)
Storage SSD with at least 100 GB free, ideally NVMe
GPU (optional) Not required unless for graphics-heavy applications
Network Gigabit LAN or stable Wi-Fi for RDP connections
BIOS/UEFI Virtualization enabled (Intel VT-x / AMD-V)
📦 Virtual Machine Configuration

Resource Minimum (for Windows Server 2019/2022)
RAM (VM) 4 GB (better: 6–8 GB)
CPU (VM) 2 virtual CPUs
HDD (VM) 50–80 GB dynamically allocated
Network Mode Bridged (for LAN-based RDP access)
⚡ What else is important?

SSD is mandatory – HDDs are too slow for virtual systems.

Cooling system: Ensure good airflow for continuous operation.

Power consumption: Mini-PCs (e.g., Intel NUC, Ryzen Mini PCs) are great if quiet and energy-efficient operation is desired.

🧪 Practical Tip
For small companies with 1–2 simultaneous RDP users, a low-cost mini-PC with Ryzen 5, 16 GB RAM, and 1 TB SSD is often sufficient. Windows Server runs stably in VirtualBox on such a system.
