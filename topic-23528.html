Hola a todos,

Este tema me apasiona. No se porque muchos no consideran el uso de xbScript, que entre las muchas ventajas me ha permitido hacer aplicaciones muy profesionales y productivas, solidas, aprovechando cualquiera de los navegadores.

Hoy día existe una alta demanda de desarrollo de aplicaciones web, pues permite abrir a un tema importante: la oficina virtual (inclusive corriendo en Blackberry, android, etc.)

xBScript es gratis, y permite que no tengas que aprender javascript (que por ciero tampoco es dificil), pues utilizas lenguaje xBase en la web. En todo caso, siempre tendras que conocer lo básico (o un poco mas, con una curva de aprendizaje en la cual tienes muchos recursos en la red para aprender) de HTML.

xBScript entra dentro de la solución que se ha planteado mas arriba de ASP, y hasta ahora SOLAMENTE en servidores Windows (utilizando IIS). Sobre Wondows 2003 va muy bien, y ni hablar de Wondows 2008 server (esto es lo único que he encontrado como "contra" al uso de xBScript)

A través de objetos OLE accesas a SQL (con ADO). Lo he probado con motor de MySQL y MS-SQL Server 5 sin problemas y realmente MUY rápido.

Inclusive, si requieres hacer "cosas" que esto no te permite, puedes desarrollar xBScript en conjunto ("conviviendo") con javascript: en uno de los proyectos necesitaba manejar gráficos, e integré la aplicación xBScript con jquery para esto, y funcionó de maravillas.

Por último, una de las aplicaciones necesitaba interactuar con Google Maps, y el desarrollo con xBScript, integrando con javascript fue muy bien integrando con "Google Maps API", que es todo un mundo para aplicaciones web que integran Google Maps.

De nuevo, me gustaría conocer las desventajas de porque no muchos utilizan xBScript (muchos desarrolladores de Fox lo utilizan ya).

Gente;

  1. Con ADS, el servidor puede ser Linux, Windows, o Novell (aunque ya nadie usa Novell).

  2. La pag puede usar xbscript, php, java, harbour cgi,....

  3. Puedes tener web portal accediendo la misma data que tus harbour win32 apps. O sea, Win32 apps, Linux apps y web portals todas accediendo tus mismos dbfs/cdx/ntx/adt/adi database a la misma vez.

  4. Puedes tener un win32 app (presumo harbour app) accediendo la data en un server remoto via conexión de internet (WAN). Cambias el tipo de conexión (en un .ini externo a la app) y sin recompilar conectas al banco de datos en el LAN. Es decir, la misma aplicación puede usar datos del WAN o del LAN sin recompilar a la vez que otros clientes pueden estar usando un browser para trabajar en la misma data con páginas escritas en php/Java/xbscript...

¿Como pudiera ser mas sencillo y con menor curva de aprendizaje que con ADS server?

Desventajas:
1. ADS no es gratis. Es muchísimo menos caro que MS-sql, pero no es gratis como MySql. ... y claro, MySql es gratis hasta que necesitas apoyo.
2. Los desarrolladores de harbour están adaptando y desarrollando hacia otros SQLs mientras que el rddads languidece.
