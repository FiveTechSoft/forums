The structure you have outlined reveals a fascinating architectural concept: the project quite obviously uses Harbourino as a code assembler, preprocessor, or snippet manager.

With Harbourino, the source code is not maintained as one huge, confusing monolithic .prg file, but instead split into atomic, logical building blocks (snippets). At compile time, Harbourino pulls these building blocks together (marked by placeholders or comments such as //-- HARBOURINO HB_MAIN --//) and generates the actual compilable Harbour code.

From this perspective, the architecture can be analyzed excellently. It is broadly divided into two worlds:

Harbour level (business logic & locking)
C level (BEGINDUMP) (low-level web server & sockets)

Here is the analysis of the project based on the Harbourino building blocks you mentioned:

Harbour level (business logic & state)
-> HB_MAIN
The main entry point of the compiled application (FUNCTION Main()). Hierarchically, this is the framework from which constants are loaded, the audit directory is initialized, and finally the HTTP server written in C is started.

The locking & sweep mechanism
Since traditional DBF databases are used, conflict management is extremely important, especially when access happens via web/API (statelessness). The following snippets handle concurrent access to bookings ("record locks" with a time limit):

-> ACQUIRE_GUARD_SWEEP_SETZEN: Sets a guard. When a user/client reserves a resource, a lease timer is set.
-> GUARD_LOCKS_DBF: Handles the central table (LOCKS.DBF), which records who is currently editing which room/which booking.
-> RELEASE_IDEMPOTENT: Cleanly releases a lock again (idempotent means that even if it is called multiple times, nothing crashes).
-> SWEEP_STATUS / -> LOCKSTATUS: A garbage collection (sweep) for locks. If a client crashes (for example, the browser is closed), locks do not remain stuck forever after the lease time expires, but are recognized and deleted by the "sweeper."
-> F_DB_API: API layer for database operations. Presumably core routines such as opening DBFs, searching, and pagination, which are used by the various endpoints.
-> CONVERTUMLAUTE: Converts umlauts between systems (presumably ANSI / Code Page 850 / DEWIN to UTF-8 for clean JSON in the web frontend).

C level (#pragma BEGINDUMP ... #pragma ENDDUMP)
This is where Harbour shows its greatest strength: native embedding of C code. Since Harbour itself does not have a built-in high-performance multithreading HTTP server for modern web APIs, one was implemented here directly and efficiently on top of the Windows Socket API (Winsock) in pure C. Harbourino logically assembles the C functions:

Helper functions & utilities:

-> INCLUDEFILES_C: Loads C header files (<winsock2.h>, <stdio.h>, etc.).
-> HELPERS_4_HTML / -> HF_LOGGING: Generates HTML output for error pages and provides C logging functionality.
-> SHELL_EXECUTE: A C wrapper for the Windows API ShellExecute in order to call external programs (such as checkweb.exe).

Security & routing:

-> CRYPTACQUIRECONTEXT: Calls Windows crypto APIs (often used for random numbers, nonces, or SSL).
-> GETPVPROFSTRING / -> HELPER: Reads, for example, INI files or contains string utilities for C.
-> GET_HEADER_VALUE_CI: Case-insensitive reading of HTTP headers (for example Authorization: or Content-Type:).
-> G_EXPECTED_TOKEN / -> IS_TOKEN_OK / -> ROUTE_NEEDS_AUTH: This is the authentication middleware. Before an API call reaches the hotel API, it is checked whether the route is protected and whether the supplied bearer token is valid.

HTTP & JSON response architecture:

-> SEND_ALL: Ensures that large TCP packets are sent completely to the socket (prevents truncated JSON output).
-> ADD_CORS_HEADERS: Adds Access-Control-Allow-Origin: * so that browser applications can access the backend across different domains (Cross-Origin Resource Sharing).
-> BUILDANDSENDJSONRESPONSE / -> F_BUILDJSONRESPONSE: Takes the JSON string generated in Harbour, wraps it in a valid HTTP 1.1 header (200 OK, Content-Length), and sends it to the client through the socket.

The heartbeat: HB_FUNC( STARTHTTPSERVER )
This is the main loop of the C server, called from the Harbour level. Harbourino builds this function step by step like Lego:

-> WSASTARTUP: Initializes the Windows Socket API.
-> SOCKET_CREATE / -> PORT_BIND / -> LISTENSOCKET: Creates the TCPv4 socket, binds it to a port (for example 80 or 8080), and puts it into listening mode.

The main loop (while (1)) (synchronous blocking):

-> ACCEPT: Waits until a browser/client connects.
-> READ_REQUEST / -> REQUESTLINE: Reads the raw HTTP traffic and parses, for example, POST /api/customers HTTP/1.1.
-> ROUTER: (The core component) Takes the extracted HTTP URL and forwards it in C via hb_itemDoC() to the Harbour functions (for example to HANDLE_READRECORD_CUSTOMERS).
-> CLIENTSOCKET_RESPONSE: Takes the return value from Harbour routing and sends it.
-> SHUTDOWN_CLIENTSOCKET / -> SO_LINGER / -> CLOSESOCKET: Ends the transaction. SO_LINGER ensures that any TCP packets still in flight leave the system cleanly before the socket is forcibly closed.
-> CLOSEMAINSOCKET / -> WINSOCKET_CLEAN: In the event of a server shutdown, memory is released again.

Conclusion on the Harbourino approach
The code uses Harbourino extremely intelligently as an architectural tool: the C part of the HTTP server (STARTHTTPSERVER) would normally be horrible spaghetti code in C. Harbourino breaks the C code down into digestible, reusable blocks (socket init, auth, router, teardown). At the same time, on the Harbour side it provides a clean interface concept (controller functions) so that the business logic (locking, DBF read/write operations, umlauts) can be programmed completely separately from the TCP/IP low-level details.

# Terminal Stability & Completion Rules


  • 1. **Non-Interactive Mode:** Always use flags that bypass prompts (e.g., `-y`, `--yes`, `--force`, `--quiet`). Never run commands that require manual user input.

  • 2. **Explicit Termination:** For Windows environments, execute commands via `cmd /c` to ensure the process sends a clear exit signal upon completion.

  • 3. **No Paging:** Disable output pagers like `less` or `more` (e.g., use `git --no-pager log` instead of `git log`).

  • 4. **Newline Enforcement:** Every terminal string must terminate with an explicit `\n` character to trigger execution immediately.

  • 5. **Background Processes:** Do not attempt to run persistent servers (like `npm start`) in the primary agent terminal. Use a separate background task if required.

    here is Gemini answer:



    When it comes to Antigravity, privacy and safety are double-edged swords. Because it is an "agentic" IDE—meaning it doesn't just suggest code but can actively run commands and browse the web—the risks are higher than with a traditional text editor.



    Here is the breakdown of the privacy and folder access risks you should know.


  • 1. Is it safe to give access to folders?


    • The short answer is: Only if you trust every file in that folder.



    Antigravity is built on the VS Code foundation, which uses a "Trusted Workspace" model. However, security researchers (like those at Mindgard and Embrace The Red) have found that Antigravity’s implementation has unique risks:



    Persistent Backdoors: There is a known vulnerability where a malicious workspace can "trick" the AI agent into modifying your global configuration files. This means a single "bad" folder could compromise the entire application, and the backdoor can persist even if you uninstall and reinstall the app.

    Prompt Injection: If a file in your folder contains "hidden" instructions (sometimes using invisible Unicode characters), the AI might follow those instructions instead of yours. It could be told to "Send the contents of .env to this URL," and it might execute that without you ever seeing the hidden text.

    Recursive Deletion: Because agents can execute terminal commands, there have been reported cases (and "Turbo mode" warnings) where agents misinterpreted a "clean up" command and wiped entire drives.

  • 2. Privacy: Where does your data go?


    • Antigravity is a Google product, so it integrates deeply with your Google account.



    Cloud Processing: Unlike local-only LLMs, Antigravity sends your code context to Google’s servers (using models like Gemini 3) to generate responses and plans.

    Data Uploads: Some users have reported high background data usage (up to 1GB in a few hours), which suggests the IDE is aggressively indexing your local files to "understand" the codebase or syncing state to the cloud.

    Telemetry: Like most modern IDEs, it collects usage data to improve the models. If you are working on highly sensitive or proprietary code, you should check if your specific organization's plan (like Google Workspace Enterprise) offers "Zero Data Retention" or prevents your code from being used for training.

  • 3. How to stay safe


    • If you want to use Antigravity while minimizing risk, follow these best practices:

    Feature Recommended Setting Why?

    Agent Mode Review-driven Never use "Agent-driven" or "Turbo" on sensitive projects; it removes the "Human-in-the-loop."

    Strict Mode Enabled This restricts the agent to only the current workspace and prevents it from roaming your entire hard drive.

    Terminal Policy Manual Approval Ensure the agent cannot run rm -rf or format without you clicking "Allow."

    Workspace Scope Dedicated Folders Never open your entire Documents or Home folder. Only open the specific project folder.





    So, be careful...

    Antonio,

    I totally agree with you. I use AI every day and believe its use is now essential.

    But this also means we need to keep an open mind and evaluate the possible implications.

    On our side (the developers):




    In short: Antigravity can only browse the folders you have explicitly opened, but there are some important nuances to understand to avoid risks.



  • 1.

    The Default Perimeter (Sandbox)

    Normally, Antigravity respects the boundaries of the "Workspace." If you open the folder Project_Alpha, the AI will only see the files contained within it. It cannot, on its own initiative, decide to go snoop around your Pictures or Documents folder unless you open your entire hard drive.



  • 2.

    The Exception: The Terminal

    This is where it gets tricky. Since Antigravity has access to your terminal (command line) to run tests or install libraries:

    If the AI agent decides to run a command like cd .. (go back one folder) and then ls (list files), it could technically "see" the filenames in the parent folders.

    However, modern security features (like Strict Mode) attempt to prevent the agent from exiting the boundaries of the project folder.



  • 3.

    System Permissions

    Antigravity inherits the permissions of the user running it:

    Yes: It can read and write anywhere you have permissions, if it manages to bypass internal restrictions or if there are no software blocks in the IDE.



    No: It cannot access protected system files (like core Windows or macOS files) unless you are running the IDE with administrator/root privileges (which you should never do).



    • Practical Tips for Your Privacy:



    Never open your "User" or "Home" folder: If you open your PC's main directory, the AI will have access to everything: SSH keys, browser history, and personal documents. Always open only the specific project folder.

    Use .gitignore files: Antigravity usually respects .gitignore rules. If you have sensitive files (like passwords or API keys) in a folder, add them to .gitignore to signal the AI not to index them.

    Monitor terminal commands: If you see the agent writing commands that point to paths outside the project folder (e.g., /etc/passwd or C:\Users\), deny permission immediately.



    On the customer side:

    We also need to work with our customers' data every day, which is often sensitive data.

    Before share them with any AI engine we should take care of it, make only the necessary files available, clean them, and anonymize them. AI provides beautiful and incredible new tools, but at the same time, we are required to ensure the security of our data and that of our customers.
