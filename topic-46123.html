Web-based Cashbook – JSON version → DBF/Microservice or SQL
Posted: Mon Nov 17, 2025 01:30 AM

I started with a very simple web-based cashbook in PHP where all data was stored in a local JSON file.

Later I switched the backend to a DBF file via a microservice – and the same pattern would also work with SQL.



The nice part: the UI code did not change, only loadData().



  • 1. Original JSON-based version

    // Old: JSON file "data.json" as backend

    // {

    // "users": [...],

    // "transactions": [...]

    // }


    • Code: Select all Collapse
    </s>function loadData() {
    $file = __DIR__ . '/data.json';

<i>    </i>$data = json_decode(@file_get_contents($file), true);

<i>    </i>if (!is_array($data)) {
<i>    </i>    $data = ['users' => [], 'transactions' => []];
<i>    </i>}

<i>    </i>return $data;
}
<e>


    Everywhere else I just used:



    $data = loadData();

    $transactions = $data['transactions'];



  • 2. DBF backend via microservice (same UI)


    • Now the data lives in a DBF file (Kasse.DBF).

    Instead of reading data.json, I call a microservice endpoint xWH_hub_readdbf.php which returns JSON.



    2.1 Minimal helper to call the microservice



    Code: Select all Collapse
    </s>function hubRequest(string $endpoint, array $payload): array
{
    $url = $endpoint; // e.g. "xWH_hub_readdbf.php" in the same directory

<i>    </i>$context = stream_context_create([
<i>    </i>    'http' => [
<i>    </i>        'method'  => 'POST',
<i>    </i>        'header'  => "Content-Type: application/json\r\n",
<i>    </i>        'content' => json_encode($payload, JSON_UNESCAPED_UNICODE),
<i>    </i>        'timeout' => 30,
<i>    </i>    ]
<i>    </i>]);

<i>    </i>// For demo purposes: no error handling, expect valid JSON
<i>    </i>$response = file_get_contents($url, false, $context);

<i>    </i>return json_decode($response, true) ?? [];
}<e>


    2.2 New loadData() using DBF field names 1:1


    Code: Select all Collapse
    </s>function loadData() {

<i>    </i>// DBF field names 1:1 – no mapping layer in PHP
<i>    </i>$payload = [
<i>    </i>    'databasePath' => 'x:\\xwhdaten\\datawin\\Kasse.DBF',
<i>    </i>    'start'        => 1,
<i>    </i>    'end'          => 500,
<i>    </i>    'fields'       => [
<i>    </i>        'TEXT',      // description
<i>    </i>        'EINNAHMEN', // income (N, 11,2)
<i>    </i>        'AUSGABEN',  // expense (N, 11,2)
<i>    </i>        'DATUM',     // date (D)
<i>    </i>        'USER',      // user code
<i>    </i>    ],
<i>    </i>];

<i>    </i>$response = hubRequest('xWH_hub_readdbf.php', $payload);

<i>    </i>// Adjust this key to your real microservice output: "records", "rows", "data", ...
<i>    </i>$rows = $response['records'] ?? [];

<i>    </i>// Keep the same structure the UI expects
<i>    </i>return [
<i>    </i>    'users'        => [],
<i>    </i>    'transactions' => $rows,
<i>    </i>];
}
<e>


    The HTML/JS part still works with $data['transactions'], so the UI does not care if the backend is JSON, DBF, or anything else – as long as the JSON shape is the same.



    3. Same pattern with an SQL-based microservice



    Architecturally, you can do exactly the same with an SQL backend:

    the PHP app doesn’t talk to the database directly; it calls a microservice that runs an SQL query and returns JSON.



    Example: microservice endpoint xWH_hub_sql_select.php:


    Code: Select all Collapse
    </s><i>
</i>function loadData() {

<i>    </i>// The microservice will execute this SQL on its side
<i>    </i>$payload = [
<i>    </i>    'sql' => "
<i>    </i>        SELECT
<i>    </i>            text       AS TEXT,
<i>    </i>            income     AS EINNAHMEN,
<i>    </i>            expense    AS AUSGABEN,
<i>    </i>            booking_at AS DATUM,
<i>    </i>            username   AS USER
<i>    </i>        FROM cashbook
<i>    </i>        ORDER BY booking_at, id
<i>    </i>        LIMIT 500
<i>    </i>    "
<i>    </i>];

<i>    </i>$response = hubRequest('xWH_hub_sql_select.php', $payload);

<i>    </i>// Again: use whatever key your microservice returns
<i>    </i>$rows = $response['rows'] ?? [];

<i>    </i>return [
<i>    </i>    'users'        => [],
<i>    </i>    'transactions' => $rows,
<i>    </i>];
}

<e>

    From the UI perspective:



    Old: JSON file → $data['transactions']



    New: DBF/microservice → $data['transactions']



    Alternative: SQL/microservice → $data['transactions']



    Same structure, same rendering logic, just a different backend behind the microservice.
