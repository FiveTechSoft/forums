You're asking how to convert mouse coordinates (</s>nRow<e>, </s>nCol<e>) from a drag-and-drop operation into the correct </s>nArrayAt<e> position in an </s>XBrowse<e> control, so you can select the item at the drop location. 0-cite-0

## Answer

To select the item at the drop position in `XBrowse`, use the `SetPos()` method with the mouse coordinates. Here's the pattern:

```harbour
FUNCTION Toplu_New_KASA_DragAll(nRow, nCol, uDropInfo, oBrw)
oBrw:SetPos(nRow, nCol, .t.)
// Now oBrw:nArrayAt is set to the correct row
// and oBrw:nColSel is set to the correct column

// Your processing code here
oBrw:Refresh()
RETURN 0
```

The `SetPos(nRow, nCol, .t.)` method converts pixel coordinates to the browse position and updates both `nArrayAt` (the array index) and `nRowSel` (the visible row position). [0-cite-1](#0-cite-1)

### Real Example from Codebase

The booking sample shows this exact pattern in action: [0-cite-2](#0-cite-2)

harbour</s><i> </i>:bDropOver := &lt;|u,r,c,f| oBrw:SetPos(r, c, .t.) oBrw:SelectedCol():Value := aData[u[1], u[2]] aData[u[1], u[2]] := 0 XEval(oBrw:bChange, oBrw) oBrw:Refresh() return nil<i> &gt; </i><e>

After calling `SetPos()`, you can access:

  • - </s>oBrw:nArrayAt<e> - the array index of the selected row

  • - `oBrw:nRowSel` - the visible row position (1-based)
  • - `oBrw:nColSel` - the column position
  • - `oBrw:SelectedCol()` - the column object at the drop position

    • ## Notes

    The `bDropFiles` codeblock receives coordinates relative to the window where drag began. [0-cite-3](#0-cite-3) Starting from FWH 18.04, additional parameters `aPoint` and `oSourceWnd` are also passed, where `aPoint` contains coordinates relative to the destination window. However, `SetPos()` handles the coordinate conversion automatically, so you can use either the original `nRow, nCol` or the newer `aPoint` values.

    You're asking about an issue with </s>TXBrowse::MouseRowPos()<e> returning 0 because </s>::nDataRows<e> is being calculated incorrectly, which you suspect is related to </s>::HeaderHeight()<e> performing extra calculations.

    ## The Problem

    Looking at the `MouseRowPos()` method, it calculates the row position from pixel coordinates and returns 0 if the calculated position exceeds `::nDataRows`: [1-cite-0](#1-cite-0)

    ```harbour
    METHOD MouseRowPos( nRow ) CLASS TXBrowse
    local nRowPos, nTmp

    if nRow <= ::FirstRow()
    return 0
    endif

    nTmp := nRow - ::FirstRow()
    nRowPos := Int( nTmp / ::nRowHeight ) + 1

    if nRowPos > ::nDataRows
    nRowPos := 0
    endif

    return nRowPos
    ```

    The issue is that `::nDataRows` is calculated during the `Adjust()` method and depends on `::HeaderHeight()` and `::FooterHeight()`: [1-cite-1](#1-cite-1)

    harbour</s><i> </i>METHOD RowCount() INLINE ( If( ::nRowHeight == nil, ::Adjust(),), Int( ( ::BrwHeight() - ::FirstRow() - ::FooterHeight() ) / ::nRowHeight ) )<i> </i><e>

    ## The Root Cause

    The `HeaderHeight()` method includes additional height when `lFull` parameter is `.t.`, adding `::nGetBarHeight` if `::nBarHdr > 0`: [1-cite-2](#1-cite-2)

    harbour</s><i> </i>METHOD HeaderHeight( lFull ) INLINE If( ::lHeader, IfNil( ::nHeaderHeight, 0 ) + ::nTopBarHeight + ; If( lFull == .t. .and. ::nBarHdr &gt; 0, IfNil( ::nGetBarHeight, 0 ), 0 ), ; If( ::lDrawBorderIn, 1, 0 ) )<i> </i><e>

    When `MouseRowPos()` calls `::FirstRow()`, which in turn calls `::HeaderHeight(.t.)`, it includes the extra bar height: [1-cite-3](#1-cite-3)

    However, `::nDataRows` is calculated based on the browse height minus the full header height, which may not match what `MouseRowPos()` expects.

    ## The Fix

    The issue is that `::nDataRows` needs to be recalculated when the browse is adjusted. Looking at the `Adjust()` method, `::nDataRows` should be set after all height calculations are complete: [1-cite-4](#1-cite-4)

    **Solution:** Ensure `::nDataRows` is properly calculated by calling `::RowCount()` after the browse is fully adjusted. In your drag-and-drop handler, you can force a recalculation:

    ```harbour
    FUNCTION Toplu_New_KASA_DragAll(nRow, nCol, uDropInfo, oBrw)
    local nRowPos

    // Ensure browse is adjusted
    if oBrw:nRowHeight == nil
    oBrw:Adjust()
    endif

    // Calculate row position manually
    nRowPos := oBrw:MouseRowPos(nRow)

    if nRowPos > 0
    // Move to the calculated row
    oBrw:Skip(nRowPos - oBrw:nRowSel)
    oBrw:nRowSel := nRowPos
    oBrw:Refresh()
    endif

    RETURN 0
    ```

    Alternatively, if `MouseRowPos()` consistently returns 0, you can calculate the position directly:

    ```harbour
    FUNCTION Toplu_New_KASA_DragAll(nRow, nCol, uDropInfo, oBrw)
    local nRowPos, nTmp

    if nRow > oBrw:FirstRow()
    nTmp := nRow - oBrw:FirstRow()
    nRowPos := Int(nTmp / oBrw:nRowHeight) + 1

      if nRowPos &lt;= oBrw:RowCount()
     oBrw:Skip(nRowPos - oBrw:nRowSel)
     oBrw:nRowSel := nRowPos
     oBrw:Refresh()
  endif

    endif

    RETURN 0
    ```

    ## Notes

    The `::nDataRows` data member is marked as "used internally" and is supposed to hold the actual number of visible data rows. [1-cite-5](#1-cite-5) It's calculated during painting and adjustment phases. If you're experiencing issues with `MouseRowPos()` returning 0, it's likely that `::nDataRows` hasn't been properly initialized yet, which typically happens on the first paint cycle. Calling `oBrw:Adjust()` or `oBrw:Refresh()` before processing the drop should resolve this.

    You're reporting that </s>::HeaderHeight(.t.)<e> is returning 5 when it should return a smaller value, causing </s>MouseRowPos()<e> to incorrectly return 0 because the calculated row position appears to be before the first data row.

    ## The Problem

    The issue is that `::HeaderHeight(.t.)` includes extra height from `::nGetBarHeight` when `::nBarHdr > 0`: [2-cite-0](#2-cite-0)

    harbour</s><i> </i>METHOD HeaderHeight( lFull ) INLINE If( ::lHeader, IfNil( ::nHeaderHeight, 0 ) + ::nTopBarHeight + ; If( lFull == .t. .and. ::nBarHdr &gt; 0, IfNil( ::nGetBarHeight, 0 ), 0 ), ; If( ::lDrawBorderIn, 1, 0 ) )<i> </i><e>

    When `lFull` is `.t.` and `::nBarHdr > 0`, it adds `::nGetBarHeight` to the header height. Since `::FirstRow()` calls `::HeaderHeight(.t.)`, this inflates the starting position for data rows. [2-cite-1](#2-cite-1)

    In `MouseRowPos()`, when you subtract this inflated `::FirstRow()` value from `nRow`, you get a smaller or negative `nTmp`, which then calculates to row position 0 or 1, and if your browse only has 3 rows (`::nDataRows = 3`), valid positions are 1-3. [2-cite-2](#2-cite-2)

    ## The Solution

    **Option 1: Bypass MouseRowPos() and calculate directly**

    Since `MouseRowPos()` is having issues with the header height calculation, calculate the row position yourself without relying on `::FirstRow()`:

    ```harbour
    FUNCTION Toplu_New_KASA_DragAll(nRow, nCol, uDropInfo, oBrw)
    local nRowPos, nTmp
    local nActualHeaderHeight

    // Use the base header height without the bar height
    nActualHeaderHeight := oBrw:HeaderHeight() // Call without .t. parameter

    if nRow > nActualHeaderHeight
    nTmp := nRow - nActualHeaderHeight
    nRowPos := Int(nTmp / oBrw:nRowHeight) + 1

      if nRowPos &gt; 0 .and. nRowPos &lt;= oBrw:nDataRows
     oBrw:nRowSel := nRowPos
     oBrw:Refresh()
  endif

    endif

    RETURN 0
    ```

    **Option 2: Check if ::nBarHdr is incorrectly set**

    The extra height is only added when `::nBarHdr > 0`. Check if this value is being set when it shouldn't be. You can temporarily disable it:

    ```harbour
    FUNCTION Toplu_New_KASA_DragAll(nRow, nCol, uDropInfo, oBrw)
    local nRowPos, nSaveBarHdr

    nSaveBarHdr := oBrw:nBarHdr
    oBrw:nBarHdr := 0 // Temporarily disable bar height

    nRowPos := oBrw:MouseRowPos(nRow)

    oBrw:nBarHdr := nSaveBarHdr // Restore

    if nRowPos > 0
    oBrw:nRowSel := nRowPos
    oBrw:Refresh()
    endif

    RETURN 0
    ```

    ## Notes

    The `::nGetBarHeight` is initialized during `Adjust()` to `FontHeight() + 8` [2-cite-3](#2-cite-3) , and the `::nBarHdr` value controls whether this extra height is included in the full header calculation. The painting code also uses `::FirstRow()` which includes this extra height [2-cite-4](#2-cite-4) , so the visual layout should match. If `MouseRowPos()` is returning 0, it suggests the mouse coordinates you're receiving in `bDropFiles` might not be relative to the browse window itself, or `::nBarHdr` is set when your browse doesn't actually have a bar area.
