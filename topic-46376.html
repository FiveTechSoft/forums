    • # **1. Introduction**

    FiveWin for Harbour includes native support for MariaDB/MySQL through
    the `maria_Connect()` function and a set of classes that allow working
    with relational databases easily, without requiring ODBC drivers or
    external components.

    This manual covers the practical use of MariaDB in FiveWin, based on the
    official examples included in the distribution (maria01.prg to
    maria16.prg and others).

    ## **Requirements**

  • - FiveWin for Harbour 25.x or higher

  • - MariaDB 10.x / MySQL 5.7+ server accessible locally or over the
    network
  • - Include `fivewin.ch` in the program
  • - Enable Unicode: `FW_SetUnicode( .t. )` if special characters are
    used

    • ## **Connection Architecture**

    FiveWin connects natively to MariaDB without ODBC, offering greater
    speed and simplicity. The connection object (`oCn`) is the central point
    from which all database operations are performed.

    # **2. Connecting to MariaDB**

    ## **2.1 maria_Connect() Function**

    The main function to establish a connection is `maria_Connect()`. It
    returns a connection object or NIL if it fails.

    ### **Syntax**

    harbour</s><i> </i>oCn := maria_Connect( cConnection, lShowError ) oCn := maria_Connect( cHost, cDatabase, cUser, cPassword [, nPort] )<i> </i><e>

    ### **Parameters -- Connection string format**

    Parameter Type Description

    ------------- ------ -------------------------------------------------

    cConnection C String format: host:port,database,user,password
    lShowError L .t. shows a message if the connection fails

    ### **Parameters -- Extended format (via FWCONNECT)**

    -----------------------------------------------------------------------

    Parameter Type Description

    --------------------------- ------------- -----------------------------

    cHost C Server name or IP. May
    include :port (e.g.,
    localhost:3306)

    cDatabase C Database name

    cUser C MariaDB user

    cPassword C Password

    nPort N Port (optional, default 3306)

    -----------------------------------------------------------------------

    ### **Connection Examples**

    ``` harbour
    // Compact string format
    oCn := maria_Connect( "208.91.198.197:3306,fwhdemo,user,password", .t. )

    // Separate parameters (local)
    oCn := maria_Connect( "localhost", "fwh", "fivetec1_antonio", "1234" )

    // Via FWCONNECT command (fivewin.ch)
    FWCONNECT oCn HOST cHost USER cUser PASSWORD cPassword DATABASE cDB

    // Built-in demo function
    oCn := FW_DemoDB() // connects to FiveWin demo server
    oCn := FW_DemoDB( 1 ) // alternate demo server
    oCn := FW_DemoDB( "ADO" ) // connection via ADO
    oCn := FW_DemoDB( "DLP" ) // connection via TDolphin
    ```

    ### **Verify Connection and Close**

    ``` harbour
    if ( oCn := maria_Connect( ... ) ) == nil
    MsgStop( "Unable to connect to server" )
    return nil
    endif

    // ... database operations ...

    oCn:Close() // or oCn:End()
    ```

    # **4. RowSet: Queries and Data Editing**

    The RowSet is the central object for working with data in
    FiveWin/MariaDB. It represents a set of records that can be displayed in
    XBrowse and edited directly.

    ## **4.1 Create a RowSet**

    ``` harbour
    // Full table
    oRs := oCn:RowSet( "clientes" )

    // Custom SQL
    oRs := oCn:RowSet( "SELECT * FROM clientes WHERE activo = 1" )

    // Parameterized SQL (avoids SQL injection)
    oRs := oCn:RowSet( "SELECT * FROM clientes WHERE estado = ?", { cEstado } )
    ```

    ### JOIN Example

    ``` harbour
    TEXT INTO cSql

    SELECT C.ID, C.FIRST, C.CITY, S.NAME AS STATENAME
    FROM customer C
    LEFT JOIN states S ON C.STATE = S.CODE

    ENDTEXT

    oRs := oCn:RowSet( cSql )
    ```

    ## **4.2 Display in XBrowse**

    harbour</s><i> </i>XBROWSER oRs FASTEDIT TITLE "Clients"<i> </i><e>

    ## **4.3 RowSet Methods**

    Method Description

    -------------------------------------- -------------------------------

    oRs:GoTop() Move to first record
    oRs:GoBottom() Move to last record
    oRs:Skip(n) Move n records
    oRs:FieldGet(n) Get field value by index
    oRs:FieldGet('name') Get field value by name

    oRs:Close() Close RowSet

    # **2\. Conexion a MariaDB**

    ## **2.1 Funcion maria\_Connect()**

    La funcion principal para establecer la conexion es maria\_Connect(). Devuelve un objeto de conexion o NIL si falla.

    oCn := maria\_Connect( cConexion, lMostrarError )

    oCn := maria\_Connect( cHost, cDatabase, cUser, cPassword \[, nPort\] )

    oCn := maria\_Connect( { cHost, cDB, cUser, cPassword } )

    ### **Ejemplos de conexion**

    // Forma 1: cadena compacta host:puerto,base,usuario,password

    oCn := maria\_Connect( "208.91.198.197:3306,fwhdemo,usuario,pwd", .t. )

    // Forma 2: parametros separados (servidor local)

    oCn := maria\_Connect( "localhost", "fwh", "usuario", "1234" )

    // Forma 3: via comando FWCONNECT (fivewin.ch)

    FWCONNECT oCn HOST cHost USER cUser PASSWORD cPassword DATABASE cDB

    // Forma 4: array de parametros

    oCn := maria\_Connect( { "localhost", "mibase", "root", "clave" } )

    // Forma 5: funcion de demo de FiveWin

    oCn := FW\_DemoDB() // servidor de demo

    oCn := FW\_DemoDB( "ADO" ) // via ADO

    oCn := FW\_DemoDB( "DLP" ) // via TDolphin

    // Forma 6: conexion embebida (sin servidor, solo 32 bits)

    oCn := maria\_Embedded( cDataFolder, "mibase", cLangFolder )

    if ( oCn := maria\_Connect( "host,db,user,pwd", .t. ) ) \== nil

    # **4\. RowSet: Consultas y Edicion de Datos**

    El RowSet es el objeto central para trabajar con datos en FiveWin/MariaDB. Representa un conjunto de registros cargados en memoria que puede mostrarse en un XBrowse y editarse directamente. Permite ordenacion y filtrado local sin consultas adicionales al servidor.

    ## **4.1 Crear un RowSet**

    // Tabla completa

    oRs := oCn:RowSet( "clientes" )

    // Acceso directo por nombre de tabla

    oRs := oCn:clientes // equivale a oCn:RowSet('clientes')

    // Con SQL personalizado

    oRs := oCn:RowSet( "SELECT \* FROM clientes WHERE activo \= 1" )

    // SQL con parametros (evita SQL injection)

    oRs := oCn:RowSet( "SELECT \* FROM clientes WHERE estado \= ?", { cEstado } )

    // JOIN entre tablas

    TEXT INTO cSql

    SELECT C.ID AS CustID, C.FIRST AS CustName,

    C.STATE AS ST, S.NAME AS StateName

    FROM customer C

    LEFT OUTER JOIN states S ON C.STATE \= S.CODE

    ORDER BY CUSTID

    ENDTEXT

    oRs := oCn:RowSet( cSql )

    ## **4.2 Mostrar en XBrowse**

    ## Architecture 1: Direct RowSet / RecSet

    The direct architecture is the standard FWH pattern: the desktop application connects directly to MariaDB using `maria_Connect()` and works with `RowSet` and `RecSet` objects. It is the simplest approach and has the most examples in FWH.



    ### RowSet Pattern — Normal Tables



    RowSet loads all data into memory, enabling local sorting and filtering without additional server queries. Ideal for tables with up to ~100K records.



    ```harbour
    oCn := maria_Connect( "server,mydb,user,password" )

    // Open table as RowSet (equivalent to USE table)
    oRs := oCn:RowSet( "clientes" )

    // SQL with safe parameters
    oRs := oCn:RowSet( "SELECT * FROM clientes WHERE estado = ?", { cEstado } )

    // Display in XBrowse
    XBROWSER oRs FASTEDIT AUTOSORT TITLE "Clients"

    // In-place editing
    oRs:First := "John"
    oRs:Salary := 50000
    oRs:Save() // Generates UPDATE only for changed columns

    // Export to JSON (key for APIs!)
    cJson := oRs:ToJson()
    ```

    ### RecSet Pattern — Large Tables (Millions of Rows)



    For tables with millions of records, `RecSet` uses automatic `LIMIT/OFFSET` pagination, loading only a "window" of data at a time:



    ```harbour
    // RecSet with 1000-record page
    oRs := oCn:RecSet( "custbig", 1000 )
    oRs:nLastRec := oCn:QueryResult( "SELECT COUNT(*) FROM custbig" )

    XBROWSER oRs FASTEDIT SHOW RECID TITLE "1 million records"
    ```

    ## Architecture 2: Hybrid — API/Service Layer



    >

    📄 `samples/web/webserv.prg` — `source/internal/fwmaria.prg#L1492-1494` (ToJson/RecToJson) — `source/function/dbffunc1.prg#L107` (FW_DbfToJson)



    For applications that need to serve **multiple client types** (FiveWin desktop + web browser + mobile app), the recommended architecture interposes a **service/API layer** between the interfaces and the database. This is exactly what your current PHP proxy does, but with the advantage of using Harbour directly as an HTTP server.



    ### Why an API Layer?



  • - **Security** — The database is not directly exposed to clients; only the API has credentials

  • - **Reuse** — Business logic is written once in Harbour and serves all clients

  • - **Scalability** — Multiple API instances can run behind a load balancer

  • - **Decoupling** — The frontend can change (React, Vue, Flutter) without touching the backend

  • - **Eliminate the PHP proxy** — Harbour already has `TWebServer`, you don't need PHP in between


    • ### Your Current Architecture vs Proposed



    ```
    CURRENT:
    Browser → PHP Proxy → Harbour Service → DBF

    PROPOSED:
    Browser / FiveWin / Mobile → Harbour API (TWebServer) → RowSet + ToJson() → MariaDB
    ```

    ## Harbour Microservice + JSON API



    >

    📄 `samples/web/webserv.prg` — `source/internal/fwmaria.prg#L1494` (RowSet:ToJson)



    This is the recommended pattern for new hybrid applications. It combines the power of Harbour/FWH with a JSON API that can feed any type of client.



    ### Example: REST API in Harbour with TWebServer



    ```harbour
    #include "FiveWin.ch"

    static oCn // shared MariaDB connection

    function Main()
    local oServer

    FW_SetUnicode( .t. )
    oCn := maria_Connect( "localhost,mydb,root,password" )
    oCn:lShowErrors := .t.

    oServer := TWebServer():New()

    // Define API routes
    oServer:Route( "GET", "/api/clients", {|req| ApiGetClients(req) } )
    oServer:Route( "GET", "/api/clients/:id", {|req| ApiGetClient(req) } )
    oServer:Route( "POST", "/api/clients", {|req| ApiCreateClient(req) } )
    oServer:Route( "PUT", "/api/clients/:id", {|req| ApiUpdateClient(req) } )
    oServer:Route( "DELETE","/api/clients/:id", {|req| ApiDeleteClient(req) } )

    oServer:Activate() // Listen on port 8080
    return nil

    // GET /api/clients?state=CA&order=salary
    static function ApiGetClients( oReq )
    local oRs, cWhere := "", aParams := {}

    if ! Empty( oReq:QueryParam("state") )
    cWhere := " WHERE state = ?"
    AAdd( aParams, oReq:QueryParam("state") )
    endif

    oRs := oCn:RowSet( "SELECT * FROM clients" + cWhere, aParams )

    return oRs:ToJson() // ← FWH already includes this!

    // GET /api/clients/42
    static function ApiGetClient( oReq )
    local oRs
    oRs := oCn:RowSet( "SELECT * FROM clients WHERE id = ?", ;
    { Val(oReq:Param("id")) } )
    return oRs:RecToJson()

    // POST /api/clients (JSON body)
    static function ApiCreateClient( oReq )
    local hData := hb_JsonDecode( oReq:Body() )
    oCn:Insert( "clients", ;
    "first,last,city,salary", ;
    { hData["first"], hData["last"], hData["city"], hData["salary"] } )
    return '{"status":"created","id":' + cValToChar(oCn:InsertId()) + '}'

    // PUT /api/clients/42
    static function ApiUpdateClient( oReq )
    local hData := hb_JsonDecode( oReq:Body() )
    local oRec := FWMariaRecord():New( oCn, "clients", ;
    "id = " + oReq:Param("id") )
    oRec:First := hData["first"]
    oRec:Last := hData["last"]
    oRec:Salary := hData["salary"]
    oRec:Save()
    return '{"status":"updated"}'

    // DELETE /api/clients/42
    static function ApiDeleteClient( oReq )
    oCn:Execute( "DELETE FROM clients WHERE id = ?", ;
    { Val(oReq:Param("id")) } )
    return '{"status":"deleted"}'
    ```

    ### Phase 1: DBF with Real-Time Replication to MariaDB



    Keep the DBF running but replicate every change to MariaDB in real time:



    ```harbour
    // DBF remains the primary data source
    oDbf := TDatabase():Open( nil, "clients", "DBFCDX", .t. )
    oDbf:SetReplicationServer( oCn ) // ← every change is replicated

    // The XBrowse stays the same for the user
    @ 20,20 XBROWSE oBrw DATASOURCE oDbf AUTOCOLS
    ```

    This lets you build and test web interfaces using MariaDB **while the desktop app continues using DBF**. See `mariasyn.prg` for a complete example.



    ### Phase 2: Switch to MariaDB Direct



    When confident, change the data source to RowSet:



    ```harbour
    // BEFORE: oDbf := TDatabase():Open( nil, "clients", "DBFCDX" )

    oRs := oCn:RowSet( "clients" )

    // The XBrowse adapts automatically
    @ 20,20 XBROWSE oBrw DATASOURCE oRs AUTOCOLS FASTEDIT AUTOSORT
    ```

    Build web and mobile frontends that consume your Harbour API. This eliminates the PHP proxy entirely.

    ## Final Recommendation



    ### Q1: Should you still use the direct RowSet approach?



    **Yes, for the initial migration.** RowSet is the fastest path and has 25+ examples in FWH. The DBF→RowSet equivalence is nearly 1:1. Start the migration with direct RowSet.



    ### Q2: Should you use a service/API layer for hybrid applications?



    **Yes, absolutely.** If your end goal includes web or mobile interfaces, the API layer is essential. But you don't need to implement it from day 1. The recommended path is:



  • 1. **First** — migrate from DBF to MariaDB with direct RowSet (no API)

  • 2. **Then** — add the API layer using `TWebServer` + `oRs:ToJson()`

  • 3. **Finally** — connect web interfaces (you can eliminate the PHP proxy)
