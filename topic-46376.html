1. Introduction

FiveWin for Harbour includes native support for MariaDB/MySQL through
the maria_Connect() function and a set of classes that allow working
with relational databases easily, without requiring ODBC drivers or
external components.

This manual covers the practical use of MariaDB in FiveWin, based on the
official examples included in the distribution (maria01.prg to
maria16.prg and others).

Requirements

  • FiveWin for Harbour 25.x or higher
  • MariaDB 10.x / MySQL 5.7+ server accessible locally or over the
    network
  • Include fivewin.ch in the program
  • Enable Unicode: FW_SetUnicode( .t. ) if special characters are
    used

Connection Architecture

FiveWin connects natively to MariaDB without ODBC, offering greater
speed and simplicity. The connection object (oCn) is the central point
from which all database operations are performed.

2. Connecting to MariaDB

2.1 maria_Connect() Function

The main function to establish a connection is maria_Connect(). It
returns a connection object or NIL if it fails.

Syntax

Code (harbour): Select all Collapse
oCn := maria_Connect( cConnection, lShowError )
oCn := maria_Connect( cHost, cDatabase, cUser, cPassword [, nPort] )

Parameters -- Connection string format

Parameter Type Description

------------- ------ -------------------------------------------------

cConnection C String format: host:port,database,user,password
lShowError L .t. shows a message if the connection fails

Parameters -- Extended format (via FWCONNECT)

-----------------------------------------------------------------------

Parameter Type Description

--------------------------- ------------- -----------------------------

cHost C Server name or IP. May
include :port (e.g.,
localhost:3306)

cDatabase C Database name

cUser C MariaDB user

cPassword C Password

nPort N Port (optional, default 3306)

-----------------------------------------------------------------------

Connection Examples

Code (harbour): Select all Collapse
// Compact string format
oCn := maria_Connect( "208.91.198.197:3306,fwhdemo,user,password", .t. )

// Separate parameters (local)
oCn := maria_Connect( "localhost", "fwh", "fivetec1_antonio", "1234" )

// Via FWCONNECT command (fivewin.ch)
FWCONNECT oCn HOST cHost USER cUser PASSWORD cPassword DATABASE cDB

// Built-in demo function
oCn := FW_DemoDB()          // connects to FiveWin demo server
oCn := FW_DemoDB( 1 )       // alternate demo server
oCn := FW_DemoDB( "ADO" )   // connection via ADO
oCn := FW_DemoDB( "DLP" )   // connection via TDolphin

Verify Connection and Close

Code (harbour): Select all Collapse
if ( oCn := maria_Connect( ... ) ) == nil
   MsgStop( "Unable to connect to server" )
   return nil
endif

// ... database operations ...

oCn:Close()   // or oCn:End()

The RowSet is the central object for working with data in

// Custom SQL
oRs := oCn:RowSet( "SELECT * FROM clientes WHERE activo = 1" )

1\. Introduccion

FiveWin for Harbour (FWH) incluye soporte nativo para MariaDB/MySQL a traves de la funcion maria\_Connect() y un conjunto de clases que permiten trabajar con bases de datos relacionales de forma sencilla, sin necesidad de drivers ODBC ni componentes externos.

Caracteristicas principales

  • Conexion directa al servidor MariaDB/MySQL via protocolo nativo TCP/IP

  • RowSets editables con soporte completo CRUD (Create, Read, Update, Delete)

  • RecSets paginados para tablas con millones de registros

  • Integracion total con XBrowse para visualizacion y edicion interactiva

  • Soporte Unicode (UTF-8) nativo

  • Tablas Pivot, relaciones padre-hijo, replicacion, cifrado AES, datos geoespaciales

  • Importacion/Exportacion desde/hacia ficheros DBF

  • Compatibilidad con Harbour y xHarbour, 32 y 64 bits

Requisitos

  • FiveWin for Harbour 25.x o superior

  • Servidor MariaDB 10.x / MySQL 5.7+ accesible en red o local

  • Incluir fivewin.ch en el programa

  • Habilitar Unicode: FW\_SetUnicode( .t. ) si se usan caracteres especiales

4\. RowSet: Consultas y Edicion de Datos

El RowSet es el objeto central para trabajar con datos en FiveWin/MariaDB. Representa un conjunto de registros cargados en memoria que puede mostrarse en un XBrowse y editarse directamente. Permite ordenacion y filtrado local sin consultas adicionales al servidor.

4.1 Crear un RowSet

// Tabla completa

oRs := oCn:RowSet( "clientes" )

// Acceso directo por nombre de tabla

oRs := oCn:clientes // equivale a oCn:RowSet('clientes')

// Con SQL personalizado

oRs := oCn:RowSet( "SELECT \* FROM clientes WHERE activo \= 1" )

// SQL con parametros (evita SQL injection)

oRs := oCn:RowSet( "SELECT \* FROM clientes WHERE estado \= ?", { cEstado } )

// JOIN entre tablas

TEXT INTO cSql

SELECT C.ID AS CustID, C.FIRST AS CustName,

      C.STATE AS ST, S.NAME AS StateName

FROM customer C

LEFT OUTER JOIN states S ON C.STATE \= S.CODE

Architecture 1: Direct RowSet / RecSet

📄 manual_mariadb/04_rowset.md — manual_mariadb/15_recset_rendimiento.md — source/internal/fwmaria.prg#L250-335

The direct architecture is the standard FWH pattern: the desktop application connects directly to MariaDB using maria_Connect() and works with RowSet and RecSet objects. It is the simplest approach and has the most examples in FWH.

RowSet Pattern — Normal Tables

RowSet loads all data into memory, enabling local sorting and filtering without additional server queries. Ideal for tables with up to 100K records.

Code (harbour): Select all Collapse
oCn := maria_Connect( "server,mydb,user,password" )

// Open table as RowSet (equivalent to USE table)
oRs := oCn:RowSet( "clientes" )

// SQL with safe parameters
oRs := oCn:RowSet( "SELECT * FROM clientes WHERE estado = ?", { cEstado } )

// Display in XBrowse
XBROWSER oRs FASTEDIT AUTOSORT TITLE "Clients"

// In-place editing
oRs:First  := "John"
oRs:Salary := 50000
oRs:Save()   // Generates UPDATE only for changed columns

// Export to JSON (key for APIs!)
cJson := oRs:ToJson()

RecSet Pattern — Large Tables (Millions of Rows)

For tables with millions of records, RecSet uses automatic LIMIT/OFFSET pagination, loading only a "window" of data at a time:

Code (harbour): Select all Collapse
// RecSet with 1000-record page
oRs := oCn:RecSet( "custbig", 1000 )
oRs:nLastRec := oCn:QueryResult( "SELECT COUNT(*) FROM custbig" )

XBROWSER oRs FASTEDIT SHOW RECID TITLE "1 million records"

  • Eliminate the PHP proxy — Harbour already has TWebServer, you don't need PHP in between

   oServer:Route( "DELETE","/api/clients/:id",  {|req| ApiDeleteClient(req) } )

   local oRs, cWhere := "", aParams := {}

                       { Val(oReq:Param("id")) } )

Final Recommendation

Q1: Should you still use the direct RowSet approach?

Yes, for the initial migration. RowSet is the fastest path and has 25+ examples in FWH. The DBF→RowSet equivalence is nearly 1:1. Start the migration with direct RowSet.

Q2: Should you use a service/API layer for hybrid applications?

Yes, absolutely. If your end goal includes web or mobile interfaces, the API layer is essential. But you don't need to implement it from day 1. The recommended path is:

  1. First — migrate from DBF to MariaDB with direct RowSet (no API)
  2. Then — add the API layer using TWebServer + oRs:ToJson()
  3. Finally — connect web interfaces (you can eliminate the PHP proxy)
