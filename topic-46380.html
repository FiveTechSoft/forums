Querido César,

Aqui tienes el análisis de deepwiki al respecto. Te agradezco tu opinión:

Sí, se puede implementar, pero hay un detalle importante a considerar sobre cómo funciona actualmente `SetCopies` en TPrinter.

## Estado actual de la clase TPrinter

El método `SetCopies` actual delega las copias al **driver de Windows** mediante `PrnSetCopies()`: [0-cite-0](#0-cite-0)

Esto llama internamente a la API de Windows para establecer `dmCopies` en el DEVMODE: [0-cite-1](#0-cite-1)

El problema es que cuando Windows gestiona las copias a nivel de driver, la aplicación envía el documento **una sola vez** y el driver produce N copias idénticas. La aplicación **no tiene visibilidad** sobre qué copia se está imprimiendo, porque ni siquiera participa en la duplicación.

## La clase NO tiene `nCurrentCopy` ni `nTotalCopies`

La definición actual de DATA en TPrinter no incluye esas variables: [0-cite-2](#0-cite-2)

También existe `PrnGetCopies()` que lee `dmCopies` del DEVMODE, pero sólo devuelve el número total configurado, no la copia actual: [0-cite-3](#0-cite-3)

## Consideración clave para la implementación

Tu propuesta es conceptualmente correcta, pero hay un conflicto: si `SetCopies` sigue llamando a `PrnSetCopies()` **y además** haces el loop en `PrintCopies`, obtendrás copias duplicadas (N copias del driver x N iteraciones del loop).

La solución correcta sería que `PrintCopies` gestione las copias a nivel de aplicación y **no** use `PrnSetCopies` para el driver, o bien que ponga `PrnSetCopies(1)` para que el driver sólo envíe 1 copia por iteración. Algo así:

```harbour
// Nuevas DATA
DATA nCurrentCopy INIT 0
DATA nTotalCopies INIT 1

// Nuevo método PrintCopies - gestiona copias a nivel aplicación
METHOD PrintCopies( bBlock ) CLASS TPrinter
LOCAL nCopia
// Asegurar que el driver sólo imprime 1 copia por iteración
PrnSetCopies( 1 )
::Rebuild()
FOR nCopia := 1 TO ::nTotalCopies
::nCurrentCopy := nCopia
Eval( bBlock, Self, nCopia )
NEXT
::nCurrentCopy := 0
RETURN Self
```

De esta forma, cada iteración del FOR genera un documento/página completo que pasa por `StartPage()`/`EndPage()` controlado por el bloque de código, y puedes diferenciar el contenido entre "ORIGINAL" y "COPIA".

Add nCurrentCopy tracking to TPrinter class
In the repository FiveTechSoft/fwh, modify source/classes/printer.prg to add copy-tracking functionality to the TPrinter class:

Add two new DATA members after line 145 (after ACCESS cName INLINE ::cDocument):

DATA nCurrentCopy INIT 0
DATA nTotalCopies INIT 1
Modify the existing SetCopies method (currently at line 341-343) to also store nTotalCopies:

METHOD SetCopies( nCopies ) INLINE ;
( ::nTotalCopies := nCopies, ;
PrnSetCopies( nCopies ),;
::Rebuild() )
Add a new PrintCopies method declaration in the class definition (before ENDCLASS at line 399):

METHOD PrintCopies( bBlock )
Add the method implementation after ENDCLASS (e.g., after the Rebuild method or at the end of the file, near other method implementations):

METHOD PrintCopies( bBlock ) CLASS TPrinter
LOCAL nCopia
// Set driver copies to 1 so copies are managed at application level
PrnSetCopies( 1 )
::Rebuild()
FOR nCopia := 1 TO ::nTotalCopies
::nCurrentCopy := nCopia
Eval( bBlock, Self, nCopia )
NEXT
::nCurrentCopy := 0
RETURN Self
The key design decision: PrintCopies sets PrnSetCopies(1) before the loop to prevent the Windows driver from also making copies (which would result in nTotalCopies * nDriverCopies total output). The application-level loop allows the user to customize each copy's content (e.g., "ORIGINAL" vs "COPIA 2").

Also update the documentation file docs/reference/classes/TPrinter.md:

Add nCurrentCopy and nTotalCopies to the Key Properties table
Add PrintCopies( bBlock ) to the Key Methods table with description "Iterates over copies allowing per-copy customization via code block"
