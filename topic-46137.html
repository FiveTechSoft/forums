La respuesta corta es: Depende totalmente de la librería cliente (libmysql.dll o libmariadb.dll) que estés vinculando con tu aplicación FWH, no del código fuente de FiveWin en sí.

FiveWin (y Harbour) actúan como un "puente". Si tu aplicación usa una DLL antigua (de la época de MySQL 5.x), no podrá negociar caching_sha2_password y fallará con MySQL 8.0+ (y definitivamente con 9.x).

Aquí tienes el desglose técnico de la situación y cómo solucionarlo:

1. El estado actual (FWH + MySQL 8/9)

  • mysql_native_password: Es el método "legacy". Las versiones antiguas de libmysql.dll (que muchos programadores de FiveWin siguen usando por costumbre) solo soportan este método.
  • MySQL 8.0: Lo soporta, pero no es el default.
  • MySQL 8.4: Está deshabilitado por defecto (hay que reactivarlo en el my.ini).
  • MySQL 9.x: Ha sido eliminado.
  • caching_sha2_password: Es el estándar moderno. Requiere un intercambio de claves RSA o una conexión segura (SSL).
    • Para que FWH soporte esto, necesitas una libmysql.dll versión 8.0 o superior, o el conector de MariaDB.

    • 2. ¿Soporta FWH ambos?

    Sí, pero solo si actualizas la DLL.

    FiveWin (a través de la clase TMySQL, TDolphin o ADODB) simplemente pasa las credenciales a la DLL.

    • Si usas una libmysql.dll antigua (v5.x): Solo funcionará mysql_native_password. Tu app fallará al conectar a un servidor MySQL 8+ con configuración default.
    • Si usas una libmysql.dll moderna (v8.x): Soporta ambos (native y caching_sha2).
    -----

    3. La Solución Recomendada: El Conector de MariaDB

    En la comunidad de Harbour/FiveWin, la solución estándar de facto para conectar a MySQL 8.0, 8.4 y 9.X no es usar la DLL de Oracle MySQL, sino la de MariaDB, debido a que es más compatible, ligera y maneja mejor la autenticación sin requerir instalaciones complejas de SSL en el cliente.

    Pasos para soportar caching_sha2_password en FWH:

    1. Descarga el MariaDB C Connector (archivo libmariadb.dll).
    2. Renómbralo a libmysql.dll (o configúralo en tu código para cargar libmariadb.dll).
    3. Colócalo junto a tu ejecutable (.exe).

    Esta librería es capaz de conectar a servidores MySQL modernos usando los nuevos protocolos de encriptación sin cambiar tu código fuente PRG.

    4. Solución Alternativa ("Workaround" rápido para MySQL 8.0/8.4)

    Si no puedes cambiar la DLL de inmediato y necesitas que tu sistema funcione ya con un servidor nuevo (previo a la versión 9.0), debes cambiar la configuración del usuario en el servidor para que use el método antiguo.

    Ejecuta esto en tu gestor de base de datos (HeidiSQL, Workbench):

    Code (sql): Select all Collapse
    -- Cambiar al usuario 'tu_usuario' para usar el password nativo antiguo
ALTER USER 'tu_usuario'@'%' IDENTIFIED WITH mysql_native_password BY 'tu_password';
FLUSH PRIVILEGES;

    Advertencia: Esto no funcionará en MySQL 9.0, ya que el plugin nativo ha sido eliminado. En MySQL 8.4, primero debes habilitar mysql_native_password=ON en el archivo de configuración del servidor antes de ejecutar el comando SQL.

    Resumen para tu desarrollo

    Componente¿Qué soporta?Acción requerida
    FWH (Código PRG)AgnósticoNinguna (tu código oServer:Connect(...) no cambia).
    DLL Antigua (5.x)Solo mysql_nativeFalla con MySQL 8 default. Debes actualizarla.
    DLL Nueva (8.x)AmbosFunciona, pero puede requerir dependencias SSL adicionales.
    MariaDB DLL (3.x)Ambos (Recomendado)Funciona perfecto con MySQL 8/9 y caching_sha2.

    En el caso de la dll de libmysql.dll de Oracle, en las versiones más recientes >=7.5, todos los plugins, incluido "caching_sha2_password"
    están incluidos en la propia dll, por ello el mayor tamaño de la dll > 6MB.

    En el caso de libmariadb no es así, ya que están en dll independientes, en el conector hay una carpeta plugin donde están las dll de cada plugin que se puede usar, por ejemplo "caching_sha2_password", se debe tomar y ponerla junto a la dll de mariadb.


    por ello el tamaño de la dll es menor.
